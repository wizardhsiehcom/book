GPU vs CPU：設計哲學差異
為什麼 GPU 更擅長 AI 運算？
▼

CPU 針對低延遲串行任務最佳化，核心少但強大，搭配大型快取和複雜的分支預測。GPU 則相反：大量簡單核心，針對高吞吐量平行任務設計。

矩陣乘法（AI 運算的核心）天然適合平行：C[i][j] 的每個元素都可以獨立計算，與其他元素無依賴關係。

關鍵洞察：GPU 用「多等待，多執行」的策略隱藏記憶體延遲——一個 Warp 在等待資料時，立即切換到另一個 Warp 繼續計算。
SMCUDA CoresTensor Cores
🧵
CUDA 執行模型
Grid → Block → Thread → Warp
▼

CUDA 用三層結構組織計算：Grid（所有 Block）→ Block（在同一 SM 上執行，共享記憶體）→ Thread（最小執行單位）。

真正的排程單位是 Warp（32 個 Thread）。所有 Warp 內的 Thread 同步執行相同指令（SIMT）。如果有分支發散（Warp Divergence），效率大幅下降。

實務重點：優化 CUDA 程式的首要目標是避免 Warp Divergence，以及讓記憶體存取合併（Coalesced Access）。
WarpSIMTOccupancy
💾
記憶體層次
為何記憶體頻寬才是瓶頸？
▼

從快到慢：Register（~8 TB/s）→ Shared Memory / L1（~20 TB/s）→ L2 Cache（~10 TB/s）→ HBM（3–8 TB/s）→ PCIe / NVLink（頻寬差 10–100×）。

LLM 推論是Memory Bound（記憶體瓶頸）：每次生成 Token，模型所有權重都要從 HBM 讀一遍，計算量相對少。這就是為何 HBM 容量和頻寬在推論場景如此重要。

MI300X 的 192 GB HBM3 vs H100 的 80 GB，在推論場景造成根本性的架構差異：前者能單卡裝載 70B 模型，後者需要 2 卡。

多 GPU 擴展策略
訓練 100B+ 模型需要什麼？
▼

大型 LLM 單卡裝不下，需要切分：資料平行（DDP，每 GPU 一份模型，切分 Batch）、Tensor 平行（切分矩陣）、Pipeline 平行（不同 GPU 負責不同層）。

多 GPU 效率取決於 GPU 間通訊頻寬：PCIe 5.0 的 128 GB/s vs NVLink 4.0 的 900 GB/s vs NVLink 5.0（B200）的 1.8 TB/s。NVLink 讓 All-Reduce 快 7–10 倍。

這是 NVIDIA B200 在大規模訓練領先的核心原因——不只是 FLOPS，而是 NVLink 讓多 GPU 通訊開銷幾乎消失。
NVLinkTensor ParallelNCCL

訓練和推論的瓶頸截然不同。看懂了架構再看這些數字，才能理解背後的原因。

🏋️
訓練效能：NVIDIA 大幅領先
MLPerf 測試顯示 2–3× 差距
▼

第三方 MLPerf Training v4.0 測試（2024）：NVIDIA H100 在 LLM 訓練上比 AMD MI300X 快約 2–3 倍。

MI300X 的峰值 FP8 TFLOPS（5,220）超過 H100（3,958），但軟體讓實際效能打折：cuDNN 的 Kernel 最佳化、NCCL 的通訊效率，都遠超 ROCm 對應庫。

硬體規格不等於實際效能。GPU 的「使用率」由軟體決定——CUDA 生態能讓 Tensor Core 保持高使用率，ROCm 仍有差距。

推論效能：AMD 有競爭力
Memory Bound 場景 MI300X 不落下風
▼

Together AI 測試（2024 Q4，LLaMA-3 70B，FP8，Batch 64）：MI300X ~2,800 tok/s，H100 ~2,100 tok/s，MI300X 領先 33%。

原因：70B 模型在 MI300X 上單卡可裝載（192 GB），省去跨 GPU 通訊；H100 需要 2 卡，All-Reduce 開銷拖慢速度。

推論的競爭和訓練完全不同：這裡更重要的是記憶體頻寬和容量，而不是 FLOPS。

量化精度的影響
FP16 / FP8 / INT4 差多少？
▼

精度降低讓模型變小、速度更快：FP16（基準）→ FP8（~2× 速度，極小品質損失）→ INT4 GPTQ（~3× 速度，依模型而定）。

B200 支援 FP4（業界首款），理論上能再提升 2×。但 FP4 的品質損失需要仔細驗證。

現代 LLM 推論幾乎都用 FP8 或 INT4，FP16 只用在精度要求極高的場景。

NVIDIA 的護城河不只是硬體。理解生態系的飛輪，才能判斷競爭者何時有機會打破壟斷。

🏰
NVIDIA 的生態護城河
400 萬開發者、40,000 家公司
▼

NVIDIA 的護城河由三層構成：硬體（Tensor Core、NVLink）→ 軟體堆疊（cuDNN、cuBLAS、NCCL、TensorRT）→ 生態系（400 萬+ 開發者、3,000+ GPU 加速應用）。

切換成本極高：把 CUDA 程式移植到 ROCm，需要確認所有第三方庫、重新測試效能、驗證模型品質。大型機構可能需要數個月工程人力。

飛輪效應：更多開發者 → 更多應用 → 更高需求 → 更多投入改善 CUDA → 再吸引更多開發者。這個飛輪從 2006 年轉到現在。

AMD ROCm 7.0 的進展
顯著改善但差距仍在
▼

ROCm 7.0（2025）主要改進：HipBLAS GEMM 效率 +30–40%、Flash Attention 2 接近 CUDA 水準、PyTorch ROCm 後端常見模型無需修改。

但訓練場景仍落後：RCCL vs NCCL 的效率差距、MIOpen 少數算子未最佳化、企業工程師招聘困難。AMD 選擇從推論場景切入，是更務實的策略。

ROCm 的策略轉變：不再硬拼 CUDA 訓練生態，而是在推論場景（記憶體容量優勢）找到差異化立足點。

打破 CUDA 壟斷的路徑
Triton / JAX / 自研晶片
▼

OpenAI Triton：用 Python 寫出接近 CUDA 效率的 GPU 程式，支援多個後端（CUDA、ROCm），降低遷移摩擦。

JAX / XLA：透過編譯器抽象層，同一份程式碼可執行在 GPU（CUDA/ROCm）、TPU、或自研晶片上。

自研晶片（AWS Trainium、Google TPU）：完全繞過 CUDA，用自家堆疊鎖定客戶。

這些路徑的共同邏輯：把抽象層提高，讓硬體選擇在編譯期決定，而非寫死在程式碼裡。
TritonJAX/XLA後端無關

RTX 50 系列的缺陷
DLSS 4 是亮點，供應是問題
▼

RTX 50 系列的技術突破是真實的：DLSS 4 Multi Frame Generation 可同時生成 3 幀中間幀，FP4 支援讓本地 AI 推論更快，Blackwell 架構遊戲效能提升 30–50%。

問題是：台積電 4nm 產能被資料中心 GPU 擠壓，消費級 GPU 缺貨嚴重。RTX 5080（$999 建議售價）長期在市場上以 $1,300+ 流通。

這是 AMD 2025 年的最大機運：NVIDIA 優先照顧 H100/B200 大客戶，消費級市場出現真空。

2022 年 NVIDIA 獨霸消費級市場（88% 份額），AMD 只剩 12%。2025 年，AMD 預計回升至 18–20%，創 2019 年 RX 5700 XT 之後最高。

FSR 4 vs DLSS 4
超採樣技術的差距縮小了
▼

DLSS（NVIDIA）長期是業界最佳超採樣技術，FSR 過去因品質不及而被詬病。RDNA 4 首次在硬體層面加入 AI Accelerator，讓 FSR 4 能執行神經網路版本的超採樣。

早期評測顯示 FSR 4 品質「顯著提升，接近 DLSS」。但 DLSS 4 Multi Frame Generation 的 3 幀生成仍是 NVIDIA 獨有優勢。

FSR 4 只支援 RDNA 4（RX 9070 以上），不像 FSR 3 可以向下相容。這是 AMD 推動升級的策略。
FSR 4DLSS 4XeSS

NVIDIA B200：訓練王者
FP4 + NVLink 5.0 + 成熟 CUDA
▼

B200 的三個關鍵優勢：FP4 支援（推論效能比 H100 提升 5×）、NVLink 5.0（1.8 TB/s，大規模訓練擴展性最強）、CUDA 生態繼承（所有 CUDA 程式無縫移植）。

雙晶片封裝：兩個 Blackwell Die 透過 10 TB/s 互連，對外呈現為單一 192 GB GPU。功耗高達 1,000W，需要特殊冷卻。

GB200 NVL72（72 個 B200 + 36 個 Grace CPU 組成的機架系統）是 OpenAI、Meta、Google 的主要採購目標。

AMD MI300X：推論競爭者
192 GB HBM3，大模型推論更划算
▼

MI300X 的核心優勢是 192 GB HBM3（5.3 TB/s 頻寬），讓 70B 以下模型可以單卡裝載，省去跨 GPU 通訊。在 Memory Bound 推論場景，比 H100 快約 30%。

訓練場景仍落後 H100（2–3 倍差距）。Azure ND MI300X v5 系列已 GA，適合推論密集型雲端工作負載。

選 MI300X 的最佳時機：需要在同一張卡上裝載 70B–140B 級別的模型，且工作負載是推論而非訓練。

🔵
Google TPU v5p
GCP + JAX = Gemini 的訓練平台
▼

TPU 基於 Systolic Array（脈動陣列），資料在固定方向的矩陣電路中流動，記憶體存取開銷遠低於 GPU。TPU v5p 的 BF16 TFLOPS 為 918，搭配 4,800 Gbps ICI 互連。

Gemini 系列模型使用 JAX + TPU v5p 訓練。JAX + XLA 是 TPU 的首選組合，PyTorch 也透過 torch_xla 支援。

TPU 的獨特優勢：數千顆 TPU 透過 ICI 直接串連成超大規模計算網格，不需要昂貴的 InfiniBand 交換器。

犧牲通用性換特定效率

AWS Trainium 2：在 AWS 上訓練 LLM，成本比 NVIDIA GPU 低 40%（官方宣稱）。透過 NeuronSDK 整合 PyTorch，但非標準算子支援有限。

自研晶片的真正意義：作為 NVIDIA 採購談判的籌碼，以及將特定工作負載從高價 GPU 轉移出去，而非完全替代 NVIDIA。

如何選擇加速器
決策框架
▼

大規模 LLM 訓練 → NVIDIA B200（NVLink + CUDA 成熟度）

大模型推論（70B+） → AMD MI300X（192 GB 單卡裝載）

GCP 深度整合 → Google TPU v5p（JAX 最佳搭檔）

AWS 長期推論 → Trainium 2 / Inferentia 2（成本最低）

通用訓練 + 推論 → H100 / H200（生態最成熟）

沒有最好的加速器，只有最適合特定工作負載的選擇。決策的關鍵問題：你的工作負載是 Compute Bound 還是 Memory Bound？
