# EE364a《Convex Optimization I》成書計畫

本計畫用來把 `data/EE364A/` 內的 Stanford EE364a（Convex Optimization I，Stephen Boyd）課程逐字稿，整理成一本繁體中文的技術書。逐字稿位於 `data/EE364A/transcripts/`（純文字、無 timestamp）。核心原則是：每一講都必須閱讀完整逐字稿，從第一行到最後一行，不以片段、摘要、搜尋命中或抽樣段落代替完整閱讀。

## 材料版本注意事項

- `data/EE364A/transcripts/` 的逐字稿為 **Stephen Boyd 2023** 錄影版本（檔名標示 `2023`）。
- `data/EE364A/course.md` 是 **2025–2026 夏季學期** 的課程網頁，授課者為 Logan Bell 與 Nikhil Devanathan，內含該學期的 deadline、考試日期、助教與評分比例。
- 這兩者屬於不同開課版本。**本書內容以 2023 Boyd 逐字稿為主線**；course.md 的行政資訊（deadline、staff、grading）只能標示為「2025–2026 學期版本」放入附錄，不可當成 2023 版本事實，也不可與逐字稿內容混寫。

## 目標

- 把課程內容整理成一本可在本 repo 內發展為 MkDocs book 的繁體中文書稿。
- 保留課程主線：凸集、凸函數、凸優化問題、對偶理論，到逼近與擬合、統計估計、幾何問題，再到數值線性代數、無約束／等式約束最小化與內點法。
- 將逐字稿中的口語講解轉成可閱讀、可複習、可交叉引用的書面章節。
- 以本地教材為輔助對照：`Convex Optimization`（Boyd & Vandenberghe）教科書 PDF 與 13 份章節 slides。
- 最後才補入網路上他人對 EE364a 的理解、筆記、討論或延伸資源，且只補充有助於理解課程主軸的資訊。

## 非目標

- 不直接將逐字稿全文貼成書稿。
- 不在未完整閱讀逐字稿前，先依課名、教科書章名或既有常識寫出最終章節。
- 不用教科書或外部文章取代課程逐字稿本身。
- 不為了速度跳過重複、口誤、板書、問答或示範段落；這些內容仍要讀完，再判斷是否進入書稿。
- 不把 slides、textbook、作業或考試資訊直接塞進章節主稿；這些材料先進入材料追蹤計畫，再判斷如何連到章節。
- 不在資訊不足時硬補內容。若本地沒有、使用者未提供、工具無法可靠取得，必須標記 `待補` 並請使用者提供，不可編造 lecture title、作業細節、檔案路徑、deadline、考試範圍、作者或連結。

## 資訊不足處理規則

本專案允許佔位，不允許猜測。逐字稿檔名只標「Lecture N」，**不含講題**，因此每一講的實際主題必須讀完逐字稿後才能確認。遇到下列狀況時，主控 agent 或 worker 必須停止補寫該欄位，改以 `待補`、`待讀後確認` 或「需使用者提供」標示：

- 講次主題尚未由完整閱讀確認（只有 Lecture 編號）。
- slides 章節與 lecture 編號無法一對一對應。
- deadline、考試範圍、grading 比例等 2025–2026 行政資訊與 2023 逐字稿版本混淆風險。
- 教科書頁碼、定理編號、習題編號未經核對。

處理方式：

1. 在 plan 或 tracker 中保留佔位。
2. 明確寫出缺少什麼材料或需要哪一講讀完才能確認。
3. 若需要使用者協助，直接提出清單。
4. 不因為想保持表格完整而自行補出不存在的資料。

## 工作階段

### 多 agent 執行制度

本書可由一個主控 agent 管控進度，再派多個章節 worker agent 逐講完成內容。主控 agent 負責整體一致性，不把品質責任完全交給 worker。

#### 角色分工

| 角色 | 責任 | 可寫入範圍 |
|---|---|---|
| 主控 agent | 維護計畫、派工、審稿、更新追蹤表、統一術語、處理跨章整合 | `plan/`、`docs/ee364a-convex-optimization/`、`configs/ee364a-convex-optimization.yml` |
| 章節 worker agent | 完整閱讀指定逐字稿，產出該講閱讀筆記與章節初稿 | 僅限被指派的 `notes/lecture-XX-*.md` 與 `XX-*.md` |
| 外部補充 agent | 在全部逐字稿初稿完成後搜尋外部理解與補充資料 | `appendix-references.md` 與指定章節的「外部補充」段落 |
| 審稿 agent | 檢查單章是否忠於逐字稿、是否跳讀、是否需要回補 | 原則上只回報問題，不直接改稿；必要時由主控 agent 整合 |

#### 派工原則

- 不一次派出 18 個章節 worker；每批建議 3 到 4 講，降低術語漂移與整合成本。
- 每個 worker 只負責 1 講；若該講很短且相鄰主題高度相關，最多可負責 2 講。
- worker 必須被告知：repo 內可能有其他 agent 或使用者的改動，不可 revert 或覆蓋非自己負責的檔案。
- worker 的任務必須具體包含：
  - 指定逐字稿完整路徑。
  - 指定閱讀範圍為第 1 行到最後一行。
  - 指定輸出閱讀筆記檔與章節檔。
  - 禁止網路搜尋，除非主控 agent 明確進入外部補充階段。
  - 禁止只根據課名、教科書章名或片段搜尋結果寫章節。
  - 若材料不足，必須回報缺口，不可自行腦補。

#### Worker 交付格式

每個章節 worker 完成後必須回報：

1. 已完整閱讀的逐字稿檔名與行數。
2. 新增或修改的檔案清單。
3. 本講 5 到 10 個核心概念。
4. 本講實際講題（讀完後確認，用來回填 tracker 的「課程主題」欄）。
5. 與前後章需要主控 agent 注意的連結。
6. 不確定或需要主控 agent 複查的點。

#### 主控 agent 驗收條件

主控 agent 收到 worker 結果後，必須檢查：

- 閱讀筆記是否明確標示完整閱讀範圍。
- 章節是否把逐字稿內容抽象成書面解釋，而不是逐字翻譯或鬆散摘要。
- 是否跳過講者板書推導、問答、例子等看似旁支但可能重要的內容。
- 術語是否和既有章節一致（凸集、凸函數、對偶、KKT、內點法等）。
- 是否新增未經允許的外部資料。
- 是否只改動被指派的檔案。
- 是否把不確定或缺失材料標成 `待補`，而不是寫成已確認事實。

驗收後主控 agent 才能更新 `plan/ee364a/ee364a-transcript-tracker.md` 的狀態。

### 第 0 階段：建立書籍骨架

1. 建立全書工作目錄，建議使用 `docs/ee364a-convex-optimization/`。
2. 建立章節命名規則、頁面模板、術語表、參考資料頁。
3. 先用 Lecture 清單建立暫定章節，不寫實質內容，只標註每章來源逐字稿；主題欄先標 `待讀後確認`。
4. 建立閱讀紀錄表，追蹤每份逐字稿是否已「完整讀完」、「抽象完成」、「章節補寫完成」、「外部補充完成」。

### 第 1 階段：課程級抽象

在逐章細讀前，先建立高層架構，但只能以課名、教科書章節與 slides 清單做粗略規劃：

1. 定義全書讀者：具備線性代數（如 EE263）與機率基礎，想系統理解凸優化的辨識與求解。
2. 依教科書結構把 18 份逐字稿粗分為幾個篇章（見下方「暫定全書篇章」，標為暫定，讀完後修正）。
3. 為每篇寫一段暫定導言，之後根據完整閱讀結果修正。

### 第 2 階段：逐字稿完整閱讀

每一講按順序處理，流程固定：

1. 開啟 `data/EE364A/transcripts/` 內對應的逐字稿 `.txt`。
2. 從頭到尾完整閱讀，不跳段。
3. 邊讀邊記錄：
   - 本講主問題
   - 關鍵概念與定義
   - 公式、定理、演算法、板書推導
   - 講者使用的直覺、例子、幾何圖像、應用
   - 問答或容易被忽略的細節
   - 與其他講次的關聯
4. 讀完後才產出該講的「抽象筆記」，並回填該講實際講題。
5. 抽象筆記完成後，再改寫成書稿章節。

### 第 3 階段：章節寫作

每章建議結構：

1. 章節導讀：這一章解決什麼問題。
2. 核心概念：用書面語整理逐字稿中的主要內容。
3. 定義與定理：整理凸集／凸函數／對偶／最優性條件等關鍵結果，必要時補幾何直覺圖。
4. 求解與應用：課程如何把問題化成可解形式、CVXPY 建模或應用場景。
5. 常見誤解：逐字稿中講者特別澄清、問答中出現、或容易混淆的點。
6. 與前後章的連接。
7. 小結：用 5 到 10 個條列收斂本章。

### 第 4 階段：全書整合

1. 統一術語翻譯（convex set、convex function、duality、KKT、interior-point、SDP、least squares 等）。
2. 檢查重複概念是否需要前置說明或交叉引用。
3. 補上全書導論與結語。
4. 建立圖表清單：凸集示意、對偶關係、KKT、內點法路徑等。
5. 檢查章節順序是否忠於課程。

### 第 5 階段：教材與課程資源整合

EE364a 的本地材料包含教科書 PDF、13 份章節 slides，以及 course.md 的作業／考試／評分資訊。這些以「關聯層」方式整合，不打散混入章節主線。由 `plan/ee364a/ee364a-materials-plan.md` 管理，原則：

1. 先建立材料清單與 slides ↔ lecture 對應（讀完逐字稿後才能確認對應）。
2. 每一講章節只保留簡短的「相關教材／作業」佔位。
3. 教科書頁碼、定理、習題編號需核對後才寫。
4. 2025–2026 行政資訊（deadline、grading、考試）只放附錄並標學期版本。

### 第 6 階段：網路補充

所有逐字稿完成閱讀與初稿後才進行外部搜尋。目標：官方 EE364a 課程頁、講義、CVXPY 文件、學生筆記、與課程主題直接相關的教科書勘誤或延伸。補充原則：

- 外部內容只補強理解，不取代逐字稿。
- 每則外部補充標註來源 URL、作者、存取日期。
- 外部說法與逐字稿不同時，保留差異並說明脈絡。

### 第 7 階段：出版化整理

1. 內容轉入 `docs/ee364a-convex-optimization/`。
2. 建立 `configs/ee364a-convex-optimization.yml`。
3. 補首頁、目錄、術語表、參考資料。
4. 執行 MkDocs build。
5. 檢查中文排版、數學式、Mermaid 圖、連結、章節導覽。

## 品質檢查清單

- 每一章都能追溯到一份完整讀完的逐字稿。
- 閱讀紀錄表不可只標「完成」，必須包含完成日期與摘要檔連結。
- 每一章的重點不是單純翻譯，而是把課程內容改造成讀者能建立心智模型的書面解釋。
- 重要定義、定理、演算法不可只用口語描述，必要時補圖或步驟。
- 外部資料必須有引用紀錄。
- 全書術語一致。

## 暫定全書篇章

依教科書結構做的**暫定**分篇；18 講與教科書章節不必一對一，實際講題以完整閱讀確認後修正。

| 篇 | 暫定章節 | 目的 |
|---|---|---|
| 第一篇：凸優化基礎 | Lecture 1-4 | 導論、凸集、凸函數、凸優化問題的辨識。 |
| 第二篇：對偶理論 | Lecture 5-6 | Lagrange 對偶、弱／強對偶、KKT 條件。 |
| 第三篇：應用問題 | Lecture 7-9 | 逼近與擬合、統計估計、幾何問題等應用。 |
| 第四篇：演算法基礎 | Lecture 10-13 | 數值線性代數背景、無約束與等式約束最小化。 |
| 第五篇：內點法與收尾 | Lecture 14-18 | 內點法與課程總結、延伸主題。 |

> 註：以上篇章與講次對應為暫定，僅供骨架；每一講讀完後回填實際講題再調整。
