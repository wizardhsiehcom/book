# CS336《Language Modeling from Scratch》成書計畫

本計畫用來把 `data/Stanford CS336 Language Modeling from Scratch/` 內的 Stanford CS336 Spring 2026 課程逐字稿，整理成一本繁體中文的技術書。核心原則是：每一講都必須閱讀完整 `.en.txt` 逐字稿，從第一行到最後一行，不以片段、摘要、搜尋命中或抽樣段落代替完整閱讀。

## 目標

- 把課程內容整理成一本可在本 repo 內發展為 MkDocs book 的繁體中文書稿。
- 保留課程的主線：從 tokenizer、PyTorch、Transformer 架構，到訓練系統、平行化、scaling laws、推論、評估、資料、post-training、alignment 與 multimodality。
- 將逐字稿中的口語講解轉成可閱讀、可複習、可交叉引用的書面章節。
- 最後補入網路上他人對 CS336 的理解、筆記、討論或延伸資源，但只補充有助於理解課程主軸的資訊。

## 非目標

- 不直接將逐字稿全文貼成書稿。
- 不在未完整閱讀逐字稿前，先依課名或既有常識寫出最終章節。
- 不用外部文章取代課程逐字稿本身。
- 不為了速度跳過重複、口誤、問答或示範段落；這些內容仍要讀完，再判斷是否進入書稿。
- 不把作業 handout、lecture code、slides、preview 或雲端算力資訊直接塞進章節主稿；這些材料應先進入材料追蹤計畫，再判斷如何連到章節。
- 不在資訊不足時硬補內容。若本地沒有、使用者未提供、工具無法可靠取得，必須標記 `待補` 並請使用者提供，不可編造 lecture title、作業細節、檔案路徑、deadline、價格、作者、連結或資源內容。

## 資訊不足處理規則

本專案允許佔位，不允許猜測。遇到下列狀況時，主控 agent 或 worker 必須停止補寫該欄位，改以 `待補`、`待查` 或「需使用者提供」標示：

- 官方材料尚未下載到 `data/`。
- 網站資訊只有文字摘錄，缺少原始 URL 或檔案。
- 課程材料名稱相似但無法判定對應講次。
- deadline、compute price、provider policy 等可能變動的資訊沒有查核日期。
- guest lecture、assignment optional part、preview、code、slides 等資訊不足以建立可靠關聯。

處理方式：

1. 在 plan 或 tracker 中保留佔位。
2. 明確寫出缺少什麼材料，例如 handout、code、preview、slides、URL 或本地路徑。
3. 若需要使用者協助，直接提出需要提供的清單。
4. 不因為章節或表格想保持完整而自行補出不存在的資料。

## 工作階段

### 多 agent 執行制度

本書可以由一個主控 agent 管控進度，再派多個章節 worker agent 逐講完成內容。主控 agent 負責整體一致性，不把品質責任完全交給 worker。

#### 角色分工

| 角色 | 責任 | 可寫入範圍 |
|---|---|---|
| 主控 agent | 維護計畫、派工、審稿、更新追蹤表、統一術語、處理跨章整合 | `plan/`、`docs/cs336-language-modeling/`、`configs/cs336-language-modeling.yml` |
| 章節 worker agent | 完整閱讀指定逐字稿，產出該講閱讀筆記與章節初稿 | 僅限被指派的 `notes/lecture-XX-*.md` 與 `XX-*.md` |
| 外部補充 agent | 在全部逐字稿初稿完成後搜尋外部理解與補充資料 | `appendix-references.md` 與指定章節的「外部補充」段落 |
| 審稿 agent | 檢查單章是否忠於逐字稿、是否跳讀、是否需要回補 | 原則上只回報問題，不直接改稿；必要時由主控 agent 整合 |

#### 派工原則

- 不一次派出 18 個章節 worker；每批建議 3 到 4 講，降低術語漂移與整合成本。
- 每個 worker 只負責 1 講；若該講很短且相鄰主題高度相關，最多可負責 2 講。
- worker 必須被告知：repo 內可能有其他 agent 或使用者的改動，不可 revert 或覆蓋非自己負責的檔案。
- worker 的任務必須具體包含：
  - 指定逐字稿完整路徑。
  - 指定閱讀範圍為第 1 行到最後一行。
  - 指定輸出閱讀筆記檔與章節檔。
  - 禁止網路搜尋，除非主控 agent 明確進入第 5 階段。
  - 禁止只根據課名、常識或片段搜尋結果寫章節。
  - 若材料不足，必須回報缺口，不可自行腦補。

#### Worker 交付格式

每個章節 worker 完成後必須回報：

1. 已完整閱讀的逐字稿檔名與行數。
2. 新增或修改的檔案清單。
3. 本講 5 到 10 個核心概念。
4. 與前後章需要主控 agent 注意的連結。
5. 不確定或需要主控 agent 複查的點。

#### 主控 agent 驗收條件

主控 agent 收到 worker 結果後，必須檢查：

- 閱讀筆記是否明確標示完整閱讀範圍。
- 章節是否有把逐字稿內容抽象成書面解釋，而不是逐字翻譯或鬆散摘要。
- 是否有跳過講者問答、作業說明、系統細節等看似旁支但可能重要的內容。
- 術語是否和既有章節一致。
- 是否有新增未經允許的外部資料。
- 是否只改動被指派的檔案。
- 是否把不確定或缺失材料標成 `待補`，而不是寫成已確認事實。

驗收後主控 agent 才能更新 `plan/cs336-transcript-tracker.md` 的狀態。

### 第 0 階段：建立書籍骨架

1. 建立全書工作目錄，建議後續使用 `docs/cs336-language-modeling/`。
2. 建立章節命名規則、頁面模板、術語表、參考資料頁。
3. 先用課程清單建立暫定章節，不寫實質內容，只標註每章來源逐字稿。
4. 建立閱讀紀錄表，追蹤每份逐字稿是否已「完整讀完」、「抽象完成」、「章節補寫完成」、「外部補充完成」。

### 第 1 階段：課程級抽象

在逐章細讀前，先建立高層架構，但只能以課名、檔案清單與課程主題做粗略規劃：

1. 定義全書讀者：已懂基本 Python/深度學習，想從零理解現代語言模型訓練。
2. 將 18 份逐字稿分成幾個篇章：
   - 基礎與工具
   - 模型架構
   - 硬體與訓練系統
   - scaling、推論與評估
   - 資料與 post-training
   - alignment、multimodality 與延伸講座
3. 為每篇寫一段暫定導言，之後會根據完整閱讀結果修正。

### 第 2 階段：逐字稿完整閱讀

每一講按順序處理，流程固定：

1. 開啟對應 `.en.txt`。
2. 從頭到尾完整閱讀，不跳段。
3. 邊讀邊記錄：
   - 本講主問題
   - 關鍵概念
   - 重要定義
   - 公式、演算法、工程限制
   - 講者使用的直覺、例子、類比
   - 問答或容易被忽略的細節
   - 與其他講次的關聯
4. 讀完後才產出該講的「抽象筆記」。
5. 抽象筆記完成後，再改寫成書稿章節。

### 第 3 階段：章節寫作

每章建議結構：

1. 章節導讀：這一章解決什麼問題。
2. 核心概念：用書面語整理逐字稿中的主要內容。
3. 從零實作角度：若要自己實作語言模型，本章告訴你什麼。
4. 工程取捨：課程提到的限制、瓶頸、設計選擇。
5. 常見誤解：逐字稿中講者特別澄清、問答中出現、或容易混淆的點。
6. 與前後章的連接：說明本章如何接到下一章。
7. 小結：用 5 到 10 個條列收斂本章。

### 第 4 階段：全書整合

逐章完成後，進行跨章整理：

1. 統一術語翻譯，例如 tokenization、attention、kernel、parallelism、scaling law、inference、evaluation、post-training、alignment。
2. 檢查重複概念是否需要前置說明或交叉引用。
3. 補上全書導論與結語。
4. 建立圖表清單：流程圖、架構圖、訓練堆疊圖、平行化策略圖、資料管線圖。
5. 檢查章節順序是否忠於課程，但必要時可在書稿中加入跨章導讀。

### 第 5 階段：作業與課程資源整合

CS336 不只有逐字稿，還包含 assignment、lecture code、slides、preview、compute guidance、honor code、submission policy 與 deadline。這些材料應以「關聯層」方式整合，而不是打散混入章節主線。

作業與資源整合由 `plan/cs336-materials-plan.md` 管理，原則如下：

1. 先建立材料清單與課程排程表，不急著下載所有檔案。
2. 每一講章節只保留簡短的「相關作業／材料」佔位，避免主稿變成作業解答。
3. 若你之後把 handout、code、slides 或 preview 下載到 `data/`，再補上本地路徑、檔案大小、用途與是否已讀。
4. assignment 內容應整理成「學習目標、實作範圍、與章節關聯」，不可產出違反課程 honor code 的解答。
5. compute provider 價格與雲端資源資訊容易過期，應只作歷史背景或另標查核日期；正式建議需重新查證。
6. honor code、AI tools policy、submission、late days、regrade 等課務規則只放在附錄或材料頁，不混入技術章節。

### 第 6 階段：網路補充

所有逐字稿完成閱讀與初稿後，才進行外部搜尋。搜尋目標：

1. 官方 CS336 課程頁、講義、作業、repo 或教材。
2. 學生或研究者的課程筆記。
3. 針對某些講次的高品質心得或解釋。
4. 與課程主題直接相關的論文、官方文件或工程文章。

補充原則：

- 外部內容只能補強理解，不取代逐字稿。
- 每則外部補充都要標註來源 URL、作者或網站、存取日期。
- 若外部說法與逐字稿不同，優先保留差異並說明脈絡，不直接混寫。
- 不收錄只有泛泛摘要、SEO 內容、無法驗證作者或與課程無關的資料。

### 第 7 階段：出版化整理

1. 將內容轉入 `docs/<book-name>/`。
2. 建立 `configs/<book-name>.yml`。
3. 補上首頁、目錄、術語表、參考資料、索引。
4. 執行 MkDocs build。
5. 檢查中文排版、Mermaid 圖、連結、章節導覽。

## 品質檢查清單

- 每一章都能追溯到一份或多份完整讀完的逐字稿。
- 閱讀紀錄表不可只標「完成」，必須包含完成日期與摘要檔連結。
- 每一章的重點不是單純翻譯，而是把課程內容改造成讀者能建立心智模型的書面解釋。
- 重要公式、系統圖、演算法或工程限制不可只用口語描述，必要時要補圖或步驟。
- 外部資料必須有引用紀錄。
- 全書術語要一致。
