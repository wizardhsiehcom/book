# 記憶體安全、整數陷阱與除錯工具

AI 系統不是因為用了 Python 或大型框架，就自動免疫於低階錯誤。相反地，只要系統裡存在 C++ extension、CUDA kernel、自訂 operator、共享記憶體或高併發服務，就一定會遇到記憶體與數值層面的風險。

## 先把錯誤分類

| 類型 | 典型結果 | AI 場景例子 |
| --- | --- | --- |
| Buffer overflow | 崩潰、資料破壞、可被利用 | 自訂 tokenizer / kernel 寫出界 |
| Use-after-free | 間歇性崩潰、鬼影 bug | tensor 或 buffer 壽命管理失誤 |
| Memory leak | 長時間變慢、OOM | 推理服務長跑後記憶體持續成長 |
| Integer overflow | 結果錯誤、UB | 量化或索引運算超界 |
| Data race | 只在高併發下出問題 | 多執行緒資料管線或快取共享 |

Binary Hacks 的價值在於，它不把這些視為「運氣不好」，而是把它們當成可以用工具捕捉的結構性問題。

## 工具鏈已經更新，但思路沒變

### 傳統工具提供的核心觀念

- Valgrind：用執行期模擬發現非法記憶體存取。
- Helgrind：用 happens-before 模型抓競態。
- gdb：把崩潰從症狀變成可定位的堆疊。

### AI 時代常見的現代對應

- ASan：快速抓 heap / stack 記憶體錯誤。
- UBSan：抓 undefined behavior，尤其整數與型別問題。
- TSan：抓資料競態。
- Compute Sanitizer：面向 GPU kernel 的記憶體與同步檢查。

工具換了，但核心方法相同：**不要猜，讓 runtime 幫你證明。**

## 整數陷阱比想像中更常見

在 AI 工程裡，整數問題常出現在最不起眼的地方：

- tensor shape 計算
- byte offset 計算
- quantization scale 與 zero-point 轉換
- 壓縮格式或索引結構的位元操作

其中最危險的不是「算錯一點點」，而是：

- 溢位後仍然通過型別檢查
- 在 release build 才出現不同結果
- 導致後續記憶體位址計算偏掉

所以 Binary Hacks 對整數、位移、符號擴展與位元遮罩的提醒，今天依然完全成立。

## AI 系統的特殊風險

### 1. Python 讓問題延後浮出水面

Python 常讓人以為記憶體安全不是自己的責任，但只要系統下探到：

- C 擴展
- C++ operator
- CUDA extension
- 第三方序列化格式

問題還是會回來，而且通常在壓力大或邊界情況才出現。

### 2. 模型檔案也是攻擊面

不是只有程式碼會出事，模型與權重格式本身也有安全選擇：

- 可執行反序列化格式，風險高。
- 可驗證、不可執行的格式，風險低。

這也是為什麼 AI 時代的「安全程式設計」不能只談 buffer overflow，還要談供應鏈與格式設計。

## 一個務實的除錯順序

1. 先確認是否能在最小輸入上穩定重現。
2. 若是原生碼，先開 sanitizer；若是 GPU，改用對應的裝置端檢查工具。
3. 把問題分類成越界、生命週期、競態、整數四類之一。
4. 再去修邏輯，而不是先重構整個系統。

## 你真正要養成的能力

不是記住每個工具的旗標，而是能快速判斷：

- 這是記憶體模型問題，還是業務邏輯問題？
- 這是 release-only bug，還是本來就存在只是平常沒被踩到？
- 這個格式或 extension 會不會把整個服務暴露在不必要的風險上？

這樣你看待 AI 系統時，就不會只盯著模型品質，而會同時保護執行品質。
