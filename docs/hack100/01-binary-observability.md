# 二進位思維與可觀察性

AI 系統最常見的錯覺之一，是把問題都看成「框架層問題」。但真實世界裡，框架只是一個外觀；底下仍然是檔案格式、位元排列、記憶體配置、符號查找與系統呼叫。Binary Hacks 的第一個重要提醒，就是**先學會看見這些東西**。

## 什麼叫做二進位思維

二進位思維不是要每個人都去手寫組語，而是養成下面這些問題意識：

- 這個檔案的格式是什麼？有沒有 header、metadata、payload？
- 這段資料是以什麼對齊方式放在記憶體中？
- 這個函式呼叫最後落到哪個符號、哪個共享函式庫？
- 失敗到底發生在「語意層」還是「載入與執行層」？

在 AI 場景中，這些問題直接對應到：

- ONNX、safetensors、TensorRT engine 的可解析性
- Python extension 與 CUDA plugin 的相依關係
- 權重載入與共享記憶體的成本
- 推理失敗時到底是模型錯、runtime 錯，還是 binary mismatch

## 可觀察性不是監控面板，而是可驗證的證據鏈

很多團隊談 observability 只想到 metrics、logs、traces。這當然重要，但對 Binary Hacks 來說，**更早的一層是二進位可觀察性**：你能不能直接檢查檔案、看懂符號、確認載入鏈、知道頁面映射與記憶體布局。

這種能力的價值在 AI 系統特別高，因為：

- 問題常跨語言：Python、C++、CUDA、driver、kernel module 同時在場。
- 問題常跨層級：框架表面只有一句錯誤訊息，真正原因藏在 ELF、ABI 或 system call。
- 問題常跨時間：冷啟動、第一個 batch、長時間運行後的碎片化，症狀都不同。

## 最小工具組

| 工具 | 你要回答的問題 | AI 場景例子 |
| --- | --- | --- |
| `file` | 這到底是什麼檔？ | 模型權重是不是你以為的格式 |
| `xxd` / `hexdump` | 前幾個位元組長什麼樣子？ | 檢查 magic bytes、版本欄位 |
| `strings` | 二進位裡藏了哪些人類可讀資訊？ | 抓出版本號、編譯器資訊、依賴痕跡 |
| `readelf` / `objdump` | 可執行檔或 `.so` 內部長什麼樣？ | 看動態依賴與符號 |
| `/proc/<pid>/maps` | 進程實際映射了什麼？ | 模型有沒有真的用 `mmap` 載入 |

## AI 時代最值得保留的習慣

1. **先看實體，再談抽象**：先確認檔案、依賴、符號、進程狀態，再討論框架邏輯。
2. **把錯誤訊息拆成層次**：載入失敗、連結失敗、初始化失敗、執行失敗不是同一件事。
3. **把每個推論建立成可驗證命題**：例如「這個模型已被 mmap 載入」應該能從 `/proc/<pid>/maps` 找到證據。

## 一個實用的檢查順序

當 AI 服務啟動異常時，先不要急著改程式碼，可以先問：

1. 檔案格式正確嗎？
2. 必要的共享函式庫都找得到嗎？
3. 進程映射是否符合預期？
4. 真的卡在模型執行，還是更早就死在載入階段？

這套順序看起來很樸素，但它會讓你把「猜問題」變成「證明問題」。
