# 第十二章：Multi-Task RL

> **逐字稿：** Lecture 12 Multi-Task RL（完整閱讀，2026-07-06）

## 導讀

本章包含兩大主題：其一是 Model-Based RL 的「合成資料生成」（延續第十一章），其二是如何訓練單一策略完成多個不同任務——即 **Multi-Task RL**。

---

## 一、MBRL：用動態模型生成合成資料

### 1.1 核心想法

已學到的動態模型不僅可用於規劃，還可以**生成合成資料**來補充真實資料。

```
完整算法（MBRL + 合成資料）：
1. 在真實環境收集資料 D_real
2. 在 D_real 上訓練動態模型 f_φ
3. 生成合成資料 D_gen：
   - 從 D_real 中採樣狀態 s
   - 從 s 開始，在學習到的模型內部做短軌跡 rollout
4. 用 D_real + D_gen 更新策略 π_θ
5. 重複
```

### 1.2 為何從中間狀態開始短軌跡？

- **從初始狀態出發**：只能覆蓋早期狀態，無法覆蓋後期狀態
- **完整長軌跡**：模型在長序列中誤差累積，可信度低
- **從任意已知狀態出發做短軌跡（推薦）**：最大覆蓋 + 最小模型誤差

```
實際效果：在真實軌跡（黑色）周圍生成很多短的探索分支（紅色）
→ 以更少真實資料達到相同策略品質
```

### 1.3 MBRL 何時值得？

| 值得 | 不值得 |
|---|---|
| 模型比策略更好學（例：迷宮——模型簡單，策略複雜）| 策略比模型更好學（例：倒水——模型需流體動力學，策略只需旋轉腕）|
| 線上資料昂貴（真實機器人、不安全環境）| 高維圖像觀測（模型學習困難）|
| 有大量無獎勵標籤的資料（模型可用，策略不行）| 模型架構超參數難調 |

## 二、Multi-Task RL：泛論

### 2.1 為什麼需要多任務 RL？

- **資料效率（均攤）**：學習一個任務的經驗可以幫助其他相似任務
- **泛化**：訓練後能泛化到新任務（zero-shot 或 few-shot）
- **實際需要**：通用助手、多技能機器人、個性化推薦系統

### 2.2 任務的形式化定義

多任務 RL = 一組 MDP $\{M_i\} = \{S_i, A_i, T_i, R_i, p_{0,i}\}$，每個任務 $i$ 有各自的獎勵函數（和可能的不同動態）。

**任務識別符（Task Identifier）$z_i$：**

| 形式 | 例子 |
|---|---|
| 整數索引 | Task 0, 1, 2 ... |
| 語言描述 | "pick up the bottle" |
| 目標狀態 | 目標圖像或向量 |
| 視頻示範 | 一段展示任務的視頻 |
| 資料集（meta 設定）| 幾條示範軌跡 |

**匯聚成單一 MDP：** 把 $z_i$ 加入狀態空間，得到 $\bar{s} = (s, z_i)$，構造出覆蓋所有任務的聚合 MDP。

## 三、多任務模仿學習

### 3.1 目標函數

$$\min_\theta \frac{1}{N} \sum_{i=1}^{N} \mathcal{L}_i(\theta) = \frac{1}{N} \sum_{i=1}^{N} \mathbb{E}_{(s,a)\sim\mathcal{D}_i}\big[-\log\pi_\theta(a|s, z_i)\big]$$

策略以 $(s, z_i)$ 為輸入，輸出動作。

### 3.2 訓練技巧

**分層采樣（Stratified Mini-Batch）：** 每個 mini-batch 確保每個任務有 $\frac{1}{N}$ 的比例。

理由：若不同任務的動作方向相反（前進 vs 後退），未分層的 batch 會讓梯度在任務間劇烈震盪，降低訓練穩定性。

**多模態架構：** 狀態（圖像）和任務描述符（語言字串）是不同模態，需要：
- 語言編碼器（Transformer）處理語言描述
- 圖像編碼器（Vision encoder）處理觀測
- 主幹網路（Backbone）整合兩者

現代做法：用視覺語言模型（VLM）統一處理。

## 四、多任務強化學習

### 4.1 架構

Q 函數和值函數也以任務描述符為輸入：

$$Q^\phi(s, a, z_i), \quad V^\phi(s, z_i)$$

多任務時常為每個任務維護獨立的 Replay Buffer（方便按任務平衡採樣）。

### 4.2 事後重標籤（Hindsight Relabeling）

**核心洞見：** 為任務 $i$ 收集的資料，可能對任務 $j$ 也有用——只需把獎勵重新標記為 $R_j$。

**算法：**
```
for 每個 episode:
  1. 採樣任務 i，以 z_i 條件執行策略，收集軌跡 (s_t, a_t, s_{t+1})
  2. 把原始資料（帶原始獎勵 R_i）存入 Buffer_i
  3. 事後重標籤：對任務 j ≠ i，計算 R_j(s_t, a_t)，構成新轉移
  4. 把重標籤資料存入 Buffer_j
5. 用所有 Buffer 更新策略
```

**重標籤啟發式：** 優先選擇 $R_j$ 高的轉移；或均勻隨機；或只對任務 $j$ 重標籤整條軌跡（軌跡中任何步驟有高獎勵都值得重標籤）。

**使用前提：**

| 條件 | 說明 |
|---|---|
| 共享狀態-動作空間 | 否則 Replay Buffer 維度不相容 |
| 相似的初始狀態分布 | 否則分享無意義 |
| 相同動態（Transition Dynamics）| 不重標籤動態，若不同則引入錯誤 |
| 不同獎勵函數 | 有不同獎勵才值得重標籤 |
| 離線（Off-Policy）算法 | 重標籤後的資料是 off-policy 的 |

### 4.3 目標條件 RL（Goal-Conditioned RL）

多任務 RL 的特殊情形：每個任務 = 達到某個目標狀態 $g$。

$$r(s, a, g) = -d(s, g) \quad \text{或} \quad \mathbb{1}[d(s,g) < \epsilon]$$

**優點：** 可自監督（用資料集中的任意狀態作為目標），不需人工定義任務。

**事後重標籤在此的形式：** 把「本來要到 $g_1$ 但實際到了 $s'$」的軌跡重標籤為「目標是 $s'$」，使失敗軌跡變成成功軌跡。

## 五、任務之間共享什麼？

| 共享 | 例子 |
|---|---|
| 動態，不同獎勵 | 同一機器人，不同動作目標 |
| 狀態空間，不同一切 | 不同遊戲（維度需一致化）|
| 語言描述中的語義結構 | 「pick up X」→ X 可泛化 |
| 平衡技能（走路、跑步、跳舞）| 所有任務共享「保持平衡」|

---

## 小結

1. **MBRL 合成資料**：從已知狀態做短 rollout，補充真實資料，不增加測試時計算量。
2. **多任務 RL** = 把任務識別符加入狀態，構造聚合 MDP，用同一套參數訓練所有任務。
3. **分層採樣**防止不同任務間的梯度對沖；**多模態編碼**處理語言+圖像輸入。
4. **事後重標籤**讓一個任務的資料重複利用於其他任務，要求共享動態、離線算法、可訪問不同任務的獎勵函數。
5. **目標條件 RL** 是多任務 RL 的特殊情形，可完全自監督。
