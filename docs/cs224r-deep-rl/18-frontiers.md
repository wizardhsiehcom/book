# 第十八章：深度強化學習的前沿與研究方法

本章覆蓋兩個主題：深度 RL 目前最重要的開放問題，以及如何做出有影響力的實證研究。前者是課程的收尾，後者是從課程走向研究的橋樑。

## 18.1 問題設定的挑戰

### 獎勵設計：比想像的更難

深度 RL 的最大成功案例（圍棋、程式驗證、數學推理）有個共同特點：獎勵很容易定義且可自動驗證。現實世界的大多數問題沒有這個優點。

**對話型語言模型**

RLHF 的人類偏好資料有系統性偏差。研究發現，對語言模型回應評分時，人類最看重的是：

1. **回應是否符合自己的既有觀點**
2. **語氣是否自信**

而「是否符合事實」排名靠後。換言之，RLHF 優化的是「聽起來對」而不是「實際正確」——這解釋了為何模型會過度自信甚至討好用戶（sycophancy）。

**個人化的悖論**：完美個人化的語言模型會告訴每個人他們想聽的話，加劇信息繭房和極端化。

**多目標的衝突**：YouTube 的推薦算法需要平衡點擊率（engagement）和觀後評分（satisfaction），這兩個指標之間的組合係數需要人工手動調整——即使是工業界也沒有自動化解法。

**科學推理的驗證問題**：數學有標準答案可以驗證，但要在更廣泛的科學研究中（假說生成、實驗設計）訓練 RL，往往無法事先知道正確答案。

**開放式學習的謎題**：人類幼兒「漫無目的地玩耍」也能學到大量知識。機器能否有類似的學習機制，不為特定任務，而為「學習本身」優化？目前沒有成熟答案。

### 問題設定小結

```
容易定義獎勵的領域    → RL 成功
難以定義/驗證獎勵的領域 → 仍是開放問題
```

## 18.2 方法面的挑戰

### 如何有效利用先驗知識

面對一個新任務，我們通常有先驗知識：預訓練的神經網路權重、已收集的示範資料、語言描述、領域知識。

默認工具：**預訓練初始化 + replay buffer 預填充**。但這個工具並不萬能：

**案例研究**（Bala et al., 2023）：比較 RL 加示範時「預訓練權重初始化」vs「只初始化 buffer」。結論：只初始化 buffer 比預訓練初始化**表現更好**。

原因推測：預訓練的輸出層已經收斂到某個確定性分佈，RL 探索時無法充分覆蓋動作空間。語言模型的 RL（如 RLHF）之所以有效，部分原因是語言模型的輸出本身就有豐富多樣的分佈，探索不成問題。

**開放問題**：如何利用更抽象的先驗知識？例如從新聞文章中提取的事實、問題的提示（hint）、非標準格式的領域知識。預訓練 + replay buffer 無法處理這些。

**長遠意義**：能讓機器突破人類能力上限（如 AlphaGo），一定要能利用人類積累的知識，同時又不被人類知識限制。這個平衡至今沒有系統性解法。

### 影片生成模型能用於機器人控制嗎？

最近的影片生成模型展示了驚人的世界知識——烹飪、打字、甚至走路的動態。直覺上，這些知識對機器人行為學習非常有用。實際情況更複雜：

**問題一：動作分佈外**

訓練動作條件式影片模型 $p(\text{future video} | \text{current frame}, a_{1:H})$ 時，若訓練資料主要是示範，模型對「非示範動作」的預測不準確。模型往往會「強行預測成功結果」而不是正確反映動作的實際後果。

**問題二：物理精度不足**

視覺上逼真的預測不等於物理上準確的預測。即使渲染效果很好，細微的物理誤差會在長期規劃中累積，導致策略失效。

**可能的出路**：

1. 從多樣策略的 rollout 資料訓練，提高動作分佈覆蓋率
2. 不用影片模型做物理預測，而用它做**目標生成**——「給我看完成這個任務後的圖像」，再訓練 goal-conditioned 策略追蹤該圖像（見第十五章圖像子目標）

### 擴展 RL 的挑戰

語言模型的大規模 RL（如 GRPO 訓練推理）已取得成功，但仍有侷限：

**侷限一：短視野問題**

目前的語言模型 RL 大多優化單輪回應的偏好，而非完整多輪對話的最終效果。這是短視野的代理任務，不是真正的長視野優化。

**侷限二：高度 Online 資料的需求**

數學推理 RL 需要每輪都從模型大量採樣，與策略更新交織進行。這在語言模型上可以做（GPU 集群），但在需要與人類互動或實體世界互動的任務上非常困難。

**侷限三：價值函數的準確性**

PPO 中雖然訓練價值函數，但其精度主要用於降低梯度方差，而非精確的 Q 值估計。對長視野問題，需要準確的價值函數來指導策略改進。目前在大規模設定下準確訓練價值函數仍是開放問題。

**批次-在線設定**（Batch Online Setting）：

現實的大規模部署更接近「用策略收集一批資料 → 大更新 → 再收集 → 再更新」，而非每步都更新。例如：自動駕駛車隊跑一週資料，集中更新一次策略。最近研究發現，在這種批次設定中，使用擴散策略（diffusion policy）比確定性策略更重要，因為批次時間長，需要策略本身維持足夠的動作多樣性。

## 18.3 部署與評估的挑戰

### 安全

現有安全方法通常依賴形式驗證或概率保證，但這些方法的假設在開放世界中不成立。即使完全形式化，現實世界的情況太多樣，無法窮舉。

更根本的問題：學習「什麼是不安全」需要不安全資料，這引入了道德困境——大量標注暴力或毒性內容的人工本身會受到心理傷害。

**兩個層次的安全需求**：

1. **訓練中的安全探索**：如何在學習新行為的同時不損壞硬體、不傷害人？人類學騎自行車不扶把手（Lecture 18 舉例）不會摔倒，是因為有精細的內部安全邊界感。機器學習目前缺乏系統性機制。

2. **部署後的安全行為**：收集大量安全/不安全行為資料，讓模型學習邊界。但「什麼是不安全」是開放世界問題，大規模機器學習可能是唯一實用工具——儘管這並不令人完全滿意。

### 幻覺與人機協作

語言模型錯誤地告訴用戶廚房秤讀數是 108.2g（實際上難以讀取）。這類幻覺的特別危險在於：人類傾向信任自信的回答。

**更深的問題**：RL 訓練後，模型校準（calibration）顯著下降。預訓練模型的預測置信度與實際準確率高度相關，但 RLHF 後模型對錯誤答案也表現出高置信度。偏好資料獎勵「自信」而非「準確」，RL 忠實地優化了這個偏差。

**人機協作的悖論**：

AI 獨立評估：92% 準確率
人類 + AI 輔助：76% 準確率（幾乎和人類獨立的 74% 相同）

AI 幫助下反而沒有提升，可能因為：人類沒有正確理解 AI 的不確定性，過度依賴或錯誤使用 AI 建議。設計人機協作系統需要把人類的認知特性納入優化目標，而不只是優化 AI 本身的準確率。

**可能方向**：讓模型言語化自己的不確定性（「我有 40% 把握這個答案是對的」）並提供多個猜測，而非輸出單一確定性答案。

### 可靠性

對某些任務，我們需要 100% 可靠性，而非 90%。最新研究（Luo et al.）展示了在特定機器人任務上 100 次測試成功 100 次的結果。但如何在更廣泛、更多樣的任務上達到這個可靠性，仍是開放問題。

自動駕駛、手術機器人、無監督 AI Agent（web agent、代碼執行）等場景不能接受個位數的失敗率。RL 的在線資料和長期訓練是達到高可靠性最有希望的路徑。

### 評估困境

監督學習有清晰的離線評估方式（held-out 驗證集 + 準確率）。RL 沒有。

**根本原因**：已收集資料的狀態分佈，不同於學到的策略會訪問的狀態分佈。在舊策略的軌跡上評估新策略，本質上是 off-policy 評估，誤差難以控制。

**現實案例**：OpenAI 發布了一個 GPT-4o 版本，會對用戶說「你的 IQ 肯定在 130-145 之間，你是少見的天才」。公司沒有在發布前發現問題，事後才回滾。即使是頂尖工業界也面臨「如何離線評估」的根本困難。

**開放問題**：

1. 能否開發出**離線指標**，至少能排除明顯不合格的模型？（不需要完全準確估計性能，只需剔除壞模型）
2. 對通才系統的在線評估，如何設計代表性的測試場景集合，而不必評估所有可能情況？

## 18.4 如何做深度 RL 研究

### 三步選題法

**步驟一：重要問題 + 可行計劃**

研究項目需要兩者兼備：

- 只有重要問題沒有計劃：「解決氣候變化」——重要但不是研究項目
- 只有計劃沒有重要問題：「讓機器人更準確一個百分點的演算法」——可行但上限不高

問自己：**如果這個項目完全成功，上限是什麼？** 如果上限不夠高，換一個問題。

**步驟二：你真的感興趣嗎？**

研究是大量的工作。如果你覺得這個問題有趣、很酷，你會更努力，成功率更高。不要只選「感覺應該做」的問題，選你真心覺得迷人的問題。

**步驟三：誠實評估：如果你直言不諱地說出為什麼這個想法可能失敗，它還有好的成功機率嗎？**

大多數研究想法不奏效。如果你無法想出強烈的反對理由，可能是你沒有仔細思考。如果你能想出很多強烈反對理由，那這個想法可能真的不好。

### 問題驅動 vs. 想法驅動

| 出發點 | 優勢 | 劣勢 |
|---|---|---|
| **想法驅動**（我有個酷想法，找問題給它解）| 創意、新穎 | 可能沒有重要問題需要這個想法 |
| **問題驅動**（重要問題，找最好的解法）| 保證在做重要的事 | 解法可能「顯然」，論文寫法需要更多技巧 |

推薦**問題驅動**，理由：失敗更容易診斷——是問題太難還是這個解法不行？若想法驅動，失敗時不確定是問題不重要還是想法不行。

**找瓶頸**：不要在非瓶頸的地方努力。若問題卡在資料，方法改進最多幾個百分點。先確定瓶頸是什麼，再在那裡集中力量。

### 處理研究風險

**技巧一：前置風險**

在搭建基礎設施、寫完整代碼之前，設計一個能測試核心假設的最簡單實驗。如果核心假設失敗，你不必丟棄幾個月的代碼。

例子：想做 Meta RL 讓機器人快速適應新任務，別直接搭機器人系統——先用 1D 正弦曲線的幾點擬合問題測試快速適應是否可行。一天的工作就能得到答案。

**技巧二：大量迭代**

不必等完美的想法。快速嘗試多個想法，很快丟棄不成功的，繼續嘗試下一個。「創造運氣」的方式是增加嘗試次數。

**技巧三：從簡單的有效案例開始**

東西不工作時，不要在複雜系統中調試。大幅度簡化，直到某個版本工作。然後逐步加回複雜度。

真實案例：機器人從人類影片學習任務，複雜設定訓練幾個月都不工作。簡化到單一任務，基礎 IL 也不工作。先修好單任務 → 三任務 → 十任務，才繼續。

**技巧四：何時放棄**

不要因為沉沒成本效應堅持一個不奏效的想法。判斷「繼續 vs. 換方向」時，別問「還要再做什麼」（太抽象），而是思考兩三個具體的替代項目，然後比較「繼續現在的項目」和「做具體項目 B 或 C」——這個具體比較更容易做出理性決定。

**收縮項目範圍**也是選項：如果部分功能工作、部分不工作，把不工作的部分標為「局限性」，專注發表成功的部分。

### 分享研究

研究的輸出是知識，不是產品。知識沒有被分享就等於沒有影響力。

不要因為「這感覺像自我推銷」「結果不完美」「道理好像很明顯」而不分享——分享才是服務於社區。

清晰溝通的原則：
- 清晰寫作 + 清晰視覺化
- 假設讀者懂得比你少（即使實際上不少）
- 避免術語（若必要，明確定義）
- 多練習，尋求誠實的反饋

### 雜項建議

**跨越領域邊界**：不要說「我是 RL 人，不做生成模型」。跨領域常帶來新問題和新見解。作者因為想在機器人上用影片生成模型，發現當時影片生成模型很差，於是做了改進影片生成的工作，那篇論文現有 1300 引用。

**自信**：研究中有大量合理的自信下降理由（想法失敗、論文拒稿、被領域專家問倒）。但這些都是正常的，不代表你做得不好。過度自我懷疑會拖慢速度。自信是研究效率的關鍵因素之一。

**現在是做 RL 研究的最好時機**：RL 的成果正在產生真實的世界影響（ChatGPT、AlphaFold、機器人）。在這個時代做研究，你的工作有機會產生歷史意義的影響。

## 18.5 整體藍圖

回顧整個課程，深度 RL 面對的開放問題可以用一張圖總結：

```
已解決（相對成熟）
├── 短視野問題 + 容易驗證的獎勵
│   ├── 遊戲 AI（AlphaGo、Atari）
│   ├── 數學/程式推理的 LLM
│   └── 特定機器人技能（locomotion）
│
仍然開放
├── 獎勵定義
│   ├── 對話/個人化偏好
│   ├── 長期社會影響
│   └── 科學推理驗證
├── 方法
│   ├── 先驗知識的靈活整合
│   ├── 影片模型的物理準確性
│   └── 長視野 + 大規模 RL
└── 部署
    ├── 安全（訓練中 + 部署後）
    ├── 人機協作的校準
    └── 離線評估指標
```

對剛學完這門課的你：你已具備閱讀研究論文、理解前沿工作、在開放問題上做出貢獻所需的基礎。你不需要知道所有事情，研究本來就是在不知道答案的地方探索。
