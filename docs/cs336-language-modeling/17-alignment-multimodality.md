# 多模態與表示對齊

## 導讀

這一講在課程排程上掛著「Alignment and Multimodality」的名字，但講者一開場就把話說清楚：原本打算延續前一講繼續談 reinforcement learning，可是整門課只剩這一堂，若完全不碰 multimodality，這門「從零打造語言模型」的課會顯得不完整——因為只要看一眼當今的主流模型，多模態幾乎無所不在。於是這一整堂課實際上是一場 multimodality 的概覽。

因此，讀者要先建立一個正確的預期：本章的「alignment」不是 post-training 那種「對齊人類偏好」（RLHF、偏好學習）的 alignment，逐字稿裡並沒有這類內容。這裡的「對齊」只有兩個技術意涵，而且都屬於多模態：一是 CLIP 用對比損失把成對的影像與文字**表示對齊**在同一空間；二是視覺語言模型（VLM）訓練中的 **alignment phase**，也就是訓練一個投影模組，把視覺編碼器吐出的向量對齊到語言模型的 token 空間。整章的主線，就是沿著這兩種「對齊」展開的。

閱讀定位上，本章應接在第 15、16 章之後，作為「課程最後把視角轉向多模態」的延伸；偏好對齊、RLHF 與 RLVR 的主線仍以前兩章為準。本章不補寫逐字稿沒有講到的 preference alignment，只保留與多模態表示相關的對齊概念。

真正的核心問題只有一個：一個天生只會吃文字 token 的 transformer，要如何處理影像、影片、音訊這些非文字模態？講者把它拆成兩半——如何把非文字資料**輸入**進 transformer，以及如何**生成**非文字資料——而這一講幾乎只回答前者，並且主要聚焦在影像。答案的第一原則貫穿全章：所有東西都得先變成 transformer 能消化的「token」，可以是離散 token，也可以是連續 embedding，但每個 token 都應該承載一個語意單位。掌握了這個原則，後面從 CLIP 到 Chameleon 的所有設計，都只是它的不同實現。

## 核心內容

### 北極星：omni-model 與「一切皆 token」

語言模型本身已經很通用：它能把任意文字映到任意文字，無論是程式碼、詩、甚至 DNA 序列。但世界是多模態的——有文字、影像、音訊、影片。一個理想的終極目標，講者稱之為 **omni-model**：能接受任意模態的任意組合作為輸入，也能輸出任意模態的任意組合。例如給它一張圖和一段影片，問關於兩者的問題；或把音訊轉成影像。本章不會真的建出完整的 omni-model，只會教其中一塊拼圖——如何把影像**輸入**進模型。

為什麼一切都繞著 transformer 打轉？因為儘管許多人嘗試別的架構，在規模化之下 transformer 在所有模態上仍是最好的選擇，於是問題就變成「如何把非文字餵給一個為文字而生的東西」。transformer 只會說 token：吃進一串 token、吐出一串 token。這一講把 token 的概念**擴張**了——它不再只是文字裡那種離散符號，也可以是連續向量（可理解為 token 的 embedding）。關鍵在於，一個 token 應該代表一個語意單位：文字的 subword 多少有語意，但單一 pixel 本身毫無意義。所以我們必須想辦法把影像、音訊都轉成離散或連續的 token。

值得一提的是，這件事在文字上其實早就做過——第一講的 BPE tokenizer 就是把文字切成 token，只是那時相對簡單。對非文字模態，我們得重新思考「BPE 的等價物」是什麼：把一張圖變成 transformer 能消化的一串東西的那個機制。由此衍生出兩個必須回答的問題：非文字資料**怎麼輸入**、**怎麼生成**。本章主攻輸入。

### CLIP：用對比損失把影像與文字對齊

把時鐘撥回 2021 年。當時語言模型已進入 foundation model 時代，視覺卻仍主要仰賴 ImageNet 這類大型人工標註資料集訓練 ResNet。OpenAI 的研究者想問：語言之所以能成功，是因為可以爬取網路上大量、雜亂但海量的文字，那影像有沒有等價物？網路上到處都是配著說明文字的圖片，能不能利用？

CLIP（contrastive language image pre-training）就是這個念頭的產物，想法乾淨俐落。給一批 image–text 對（例如一批 32,000 對），用一個 image encoder 把每張圖編成向量，用一個 text encoder 把對應文字編成向量。訓練目標是：讓**成對的**影像與文字向量的內積，遠大於這張圖和其他所有文字的內積；反過來，也讓這段文字和它對應影像的內積，大於它和其他所有影像的內積。這相當於在一個 N×N 的相似度矩陣上，逐列、逐行各做一次 softmax 分類（共 2N 個 cross-entropy），把對角線推高、把非對角線壓低。程式上就是把兩邊 embedding 正規化、乘上溫度係數、算 cross-entropy。

一些落地細節值得記住。**資料**方面，論文著墨不多，大致是拿一批查詢上網搜尋、挖出 image–text 對，最終得到 4 億對，但資料集未釋出；後來 OpenCLIP 做了開源復現，用的是 LAION-5B，號稱 50 億對，有趣的是他們**用 CLIP 本身來過濾資料**再訓練 OpenCLIP，帶點自舉的味道，但至少資料與程式碼可指認。**影像前處理**方面，網路影像尺寸五花八門，而神經網路討厭動態尺寸，所以把短邊縮到 336（或 224）再從中間裁成正方形——這是為了方便，而且當時作者腦中想的是 ImageNet 分類，物體通常在正中央，裁掉邊緣背景無妨（後面會看到更講究的做法）。

**視覺編碼器**方面，CLIP 試過 ResNet 和剛問世的 ViT，發現 ViT 最好，所以現在講 CLIP 多半指 ViT 版本。ViT 把影像切成小 patch（原始論文 16×16，CLIP 用 14×14），每個 patch 就是一個 token，加上位置編碼後過標準 transformer。要從一串輸出向量得到單一表示，CLIP 沒用單純平均，而是做 **attention pooling**：以全域平均當 query，對各位置再做一輪 attention，得到資訊更豐富的向量。最佳模型是 **ViT-L/14**（large、14×14 patch、RGB 三通道、336×336 解析度；為省算力先用低解析度訓練，後段才拉高）。至於位置編碼，他們試過 2D 版但發現對分類差別不大（存疑：對分類也許不重要，後面會有更講究空間結構的版本）。**文字編碼器**則是 GPT-2 風格的 transformer（本就是同一批人做的），前後加 BOS/EOS，取最高層 EOS 位置的 activation 當整句表示。

讓 CLIP 一炮而紅的結果是：在 ImageNet 上，**zero-shot 的 CLIP 勝過在 120 萬張標註影像上訓練的 ResNet**。那 120 萬張標註耗費了大量 Mechanical Turk 人力，而 CLIP 只用了更「有機」的網路資料，測試時把影像和各個文字標籤算內積、取最高者即可完成分類。這裡也埋了一個伏筆：CLIP 還試過另一種目標——從影像**生成** caption 文字（無論當成 bag of words 或語言模型），結果較強的生成式模型反而更差、更沒效率（以分類準確率衡量）。這暗示：要學到影像的粗略語意表示，精確建模 caption 的逐字序列並不重要。這一點在全章結尾會再度浮現。

| 路線 | 影像如何進入模型 | 訓練重點 | 能否直接生成影像 |
|---|---|---|---|
| CLIP / SigLIP | 影像 encoder 產生語意向量，與文字向量對齊 | image-text 表示對齊 | 否 |
| LLaVA / Qwen-VL | vision encoder 輸出經 projector / adapter 接到 LM token 空間 | alignment phase 與多模態指令資料 | 否，只輸出文字 |
| Chameleon | VQ-VAE 把影像離散化成 image tokens，與文字 token 混合 | 單一 LM 同時建模文字與影像 token | 是，但有資訊損失與訓練穩定性問題 |

### SigLIP：用 sigmoid 損失換取效率與彈性

CLIP 有個技術缺點：它需要很大的 batch（例如 30,000），batch 太小就完全學不動，而且 softmax 是對**整個 batch** 運算的，不像語言模型每條序列可獨立平行、最後再聚合。Google 的 SigLIP（Sigmoid Loss for Language Image Pre-Training）針對這點改良。差別在損失函數：CLIP 是多類別分類（一個對齊對 vs. 其餘所有影像或文字）；SigLIP 把它簡化成**逐對的二元分類**——對任一 image–text 對，只問「對齊還是不對齊」，在相似度矩陣裡對角線是正例、非對角線是負例，用 log-sigmoid。

這個看似微小的改動帶來兩個好處。其一是**效率與可平行性**：損失可以像系統課的資料平行那樣分散——每個裝置存一部分 image–text 對、先各自算本地損失，再把文字 embedding 沿裝置環狀輪轉，直到覆蓋所有非對角線區塊。（實測上 CLIP 用 256 顆 TPU v3 訓 10 天，SigLIP 用 32 顆 v4 訓 5 天；講者還澄清 v4 每秒浮點運算其實不比 v3 快，優勢在能塞更多顆進一個 pod、互連更好。）其二是 **batch size 與損失解耦**：CLIP 的損失綁死 batch size，改 batch 就換了一個損失函數；SigLIP 則可用更小的 batch（低於 16k），期望上是同一個損失、只是變異較大。往上加大 batch 幫助有限，critical batch size 大約在 32k。資料則用 Google 的 WebLI，約十億對，並對圖中含文字者做 OCR 形成額外配對，且是多語言。

到這裡，我們手上有了 CLIP / SigLIP 這種「把一張固定尺寸影像（如 336×336）映成一個帶語意向量」的影像編碼器。接下來就能拿它們去組裝真正的多模態模型。

### VLM 的通用模板：把影像 embedding 注入語言模型

視覺語言模型（VLM）的基本想法非常務實：拿一個現成的影像編碼器、一個現成的語言模型，把影像編出的向量**投影**後注入語言模型。這比較像 mid-training 或 post-training 的味道——縫合現成零件，而不是從零訓練。講者用 LLaVA 與 Qwen 兩大家族來說明，兩者模板相近、細節各異。

**LLaVA（2023）** 是這條路的起點。它用 CLIP 當視覺編碼器，用 Vicuna 當文字解碼器，中間用一個矩陣 W 當投影模組，把影像向量映到與文字 embedding 相同的空間。之後文字與影像的向量串成一整串，過標準 transformer 輸出——等於把影像「翻譯」成文字 token，好借用預訓練語言模型的能力。它的訓練資料是合成的：拿 MS-COCO 的標註去 prompt GPT-4，生成三類資料——對話式問答、詳細描述、複雜推理，共 15.8 萬例。訓練分兩階段：**第一階段（alignment phase）** 凍結視覺編碼器與語言模型、只訓 W，讓影像向量長得像自然語言 token embedding；**第二階段** 仍凍結視覺編碼器、同時訓 W 與語言模型。一個經典例子是模型能看出「在小廂型車後面燙衣服」是不尋常的，而且即使使用者沒特別問「哪裡奇怪」也能主動點出。

**LLaVA-OneVision（2024）** 是同一配方的升級：視覺編碼器換成 SigLIP，解碼器換成 Qwen2，投影模組從線性換成兩層 MLP，並能處理單圖、多圖與影片（影片就是抽取若干幀的影像序列）。這一代最重要的技巧是 **AnyRes**（源自 LLaVA-1.5）：由於視覺編碼器只吃 336×336，而 OCR 需要保留細節（否則 J 看起來像 I），做法是把高解析度影像切成多個 336×336 的 tile 分別編碼再串接，另加一路整張圖降採樣的全域編碼；tile 太多就降採樣或內插控制 token 數。這種自適應解析度，本質上和 transformer 早已能處理任意長度序列是同一件事——影像也可以是任意解析度。為了不讓某個模態壓過其他模態，它刻意加權：單圖給較多 tile（最多 9 個），多圖用基礎解析度，影片每幀用更少 token（最多 32 幀，否則撞上 context length）。這一代還觀察到有趣的**跨模態遷移**：訓練時只有單圖的 OCR 資料、只有多圖的關係推理資料，測試時卻能處理「一張表格加一張圖表」的多圖問答，甚至把單圖的 visual prompting（圈出某處）遷移到影片。任務夠多時，模型會自己長出遷移能力。LLaVA 系列的可貴之處是同時開源模型與資料，可供複現。

**Qwen 家族** 走的是同一模板但不斷加料。**QwenVL（2023）** 用 OpenCLIP 當編碼器，adapter 是一層 cross attention 加 2D 位置編碼、把影像壓到固定 256 個 token，並設計了 image / box / ref 等特殊 token；它的三階段訓練中，第一階段反而是**凍結語言模型、訓練視覺編碼器與 adapter**（與 LLaVA 相反），能輸出 bounding box 座標與 OCR。**Qwen2-VL** 引入 **dynamic resolution**：大圖可能對到上萬個 token、一張小小的公式圖只對到 8 個 token，做法沿用 AnyRes，並把每 2×2 patch 壓成 1 以省 context，影片每秒抽 2 幀、上限 16,000 token。它的位置編碼改用 **M-RoPE（multimodal RoPE）**：RoPE 讓向量內積只依 token 間距離，1D 的距離就是相隔幾個 token，而多維版把位置變成 (height, width, time) 三元組，各維各算 RoPE 再串接。**Qwen3-VL** 則是一連串品質導向的改良：改用很強的 Qwen3 語言模型；把 context 拉到 256K 以支援長影片；編碼器換 SigLIP2；把 M-RoPE 的頻率**交錯**分配（先前 time、width、height 各佔一整塊維度，導致時間只落在低頻、高度只落在高頻，交錯後每軸都涵蓋高低頻）；加入**顯式的影片時間戳 token**（把「0 秒」做成真正的 token，可直接指涉「兩秒後發生什麼」）；用**平方根正規化的 per-token 損失** downweight 過長的影片樣本；並用 **DeepStack** 把視覺編碼器算出的一疊 embedding 直接加進語言模型的 residual stream，做更深的融合。它的訓練管線已相當複雜：預訓練四階段（先訓 adapter，再把序列從 8k 逐步拉到 32k、256k），post-training 三階段（長 CoT 的 SFT、知識蒸餾、RL）。在 benchmark 上，Qwen3-VL 面對閉源的 Gemini、GPT-5、Opus 4.1 也相當有競爭力。

一個常被問到的細節：視覺編碼器的參數量遠小於語言模型（LLaVA 的例子裡語言模型 72B、投影模組約 72M、ViT 通常小於 1B）。原因是視覺編碼器做的是很**局部**的操作——只看一個小 patch、理解它，不做推理、幾乎不含知識；模型真正的能力仍在語言模型裡。

還要記住一件事：以上這些 VLM 全都**只生成文字**，多模態只發生在**輸入側**。除了 RL 階段，訓練時多半是逐 token 監督，模型生成什麼由資料集決定。若要讓模型也能**生成**影像，就得換一條路。

### Chameleon：把一切離散化，用單一語言模型統一處理

前面的 VLM 都把影像編成連續向量注入語言模型，因此只能生成文字。Meta 的 **Chameleon（2024）** 反其道而行：把**所有東西都映成離散 token**。這樣影像的理解與生成就能用完全相同的方式進行，還能交錯生成文字與影像——例如使用者說「我很無聊，給我看些鳥」，模型就回一段文字、幾張圖、再一段文字、再幾張圖。這正是 omni-model 願景的一種實現：讓文字與影像真的活在同一個空間，方法是讓一切都長得像文字。

技術上，它用 **VQ-VAE**（vector quantized variational autoencoder，約 2017 的舊想法）把影像離散化：編碼器先把影像映到連續向量，再四捨五入到一個 codebook 裡最接近的 code（例如 8,000 個原型向量），解碼器再從 code 重建影像，用重建損失訓練（因不可微還得加些額外項）。最終一張 512×512 的影像被轉成 1,024 個 token，每個 token 來自 8,000 大小的詞彙表。之後訓練一個新的 tokenizer，接著就是**完全普通的語言模型訓練**——沒有 adapter、沒有獨立的視覺編碼器，就只有一個語言模型。這正是它在美學上吸引人的地方：一個模型、一視同仁地對待所有模態。

但它有幾個真實的代價。其一是**訓練不穩定**：文字與影像雖然都被叫做「離散 token」、佔同一空間，行為卻天差地遠——文字 token 熵較低（多數字相對可預測），影像 token 熵很高（不知道下一格是哪個藍），混在一起訓練會讓參數的 norm 失控成長、損失不穩，得靠 QK-norm 與 Z-loss 正則化才部分壓得住。其二是**離散化會丟資訊**：小字一旦離散化就再也讀不出來，OCR 這類任務因此吃虧，整體性能也不如注入式 VLM。其三，多模態一起訓本就 tricky，這裡更嚴重。VQ-VAE 曾因「transformer 只能生成離散東西」而在影像生成上流行一時，但 diffusion model 崛起後，這條把影像塞成離散 token 的路線就沒那麼吃香了。

### 收尾反思：omni-model 可能長什麼樣

當今的 frontier 模型被預期是多模態、甚至原生多模態或 omni-model；Gemini、GPT 發表時都強調這點，而且確實做到了，只是沒有公開細節。講者明說這是他的推測：大概是「連續編碼器（避免資訊損失）加上 diffusion 生成」的組合。根本的挑戰始終是如何處理非文字模態，而且理解模態與生成模態之間存在一種對稱性，卻**沒有單一通用的編碼器**——CLIP 為了分類，只需捕捉高層語意，向量可以很小；但 OCR 或影像生成需要非常細的細節，這也是 diffusion 好用的原因，它能微調高頻資訊。實務上還要小心替各模態適當加權（影片的資訊密度低於文字，別讓它壓過文字）。而至少到目前為止，連續編碼器仍是首選：CLIP 雖已問世五年，類似想法仍是捕捉影像語意的主力；transformer 依舊在場；diffusion（本講未展開）則擅長生成。

## 工程取捨

**連續 embedding 還是離散 token？** 這是本章最核心的分岔。注入式 VLM（LLaVA、Qwen）用連續 embedding，保真度高、性能好，但語言模型天生只能生成文字，要生成影像得外接 diffusion head。Chameleon 用離散 token，換來「一個模型統一所有模態、可直接生成影像」的優雅，代價是離散化丟細節、訓練不穩定、整體性能較弱。目前風向偏向連續編碼器加 diffusion 生成。

**CLIP 的多類別對比 vs. SigLIP 的逐對二元。** CLIP 的 softmax 綁死大 batch、對整批運算、不易分解；SigLIP 用 sigmoid 把 batch 與損失解耦，可平行、可用小 batch，效率更好。當你在意訓練吞吐與彈性時，二元式對比是更工程友善的選擇。

**固定解析度 vs. 動態解析度。** 早期 CLIP 直接把影像縮裁成固定 336×336，簡單但做不了 OCR、讀不了文件。AnyRes 與 dynamic resolution 用「切 tile、各自編碼、串接」換取細節保真與對任意尺寸的適應，代價是 token 數暴增，尤其影片會撞上 context length，於是又得靠降幀、壓 patch、長 context、乃至對長樣本做損失加權來平衡。

**凍結什麼、訓練什麼。** alignment phase 只訓投影模組、凍結編碼器與語言模型，成本低且穩定；之後再分階段解凍。LLaVA 全程凍結視覺編碼器，QwenVL 第一階段卻反過來訓練視覺編碼器、凍結語言模型——沒有唯一正解，取決於你想把能力灌到哪一塊，以及資料的規模與品質。

**資料策展與蒸餾。** VLM 的多模態能力極度依賴資料工程，且大量做法是不加掩飾地蒸餾 GPT-4 這類更強的模型——在沒有標註預算時這是務實選擇，但也帶來對外部模型的依賴與潛在的授權疑慮。

## 常見誤解

**「這一講在講對齊人類偏好。」** 不是。本講的 alignment 專指多模態的表示對齊（CLIP 的對比對齊、VLM 的投影對齊），與 RLHF、偏好學習無關；講者原本要講的 RL 內容被 multimodality 取代了。

**「多模態模型會生成影像。」** 本講討論的 VLM（LLaVA、Qwen）**只生成文字**，多模態只在輸入側。要生成影像得走 Chameleon 那種離散化路線，或外接 diffusion。

**「把影像叫做離散 token，它就跟文字一樣了。」** 叫法一樣不代表行為一樣。文字 token 熵低、影像 token 熵高，硬混在一起訓練會導致 norm 成長與損失不穩，Chameleon 得靠 QK-norm 與 Z-loss 才勉強壓住。

**「視覺編碼器應該很大才對。」** 相反，它通常小於 1B，遠小於數十 B 的語言模型，因為它做的是局部的 patch 級操作、不負責推理與知識，能力主要住在語言模型裡。

**「有一個通用的最佳影像編碼器。」** 沒有。捕捉高層語意（分類）可以用小向量的 CLIP，但 OCR 與影像生成需要細粒度細節（diffusion 之所以強），需求不同、編碼器就不同。

**「更強的生成式目標一定學到更好的表示。」** CLIP 的實驗顯示，用較強的生成式模型去預測 caption，反而不如簡單的 bag-of-words 對比——要學粗略語意，逐字建模 caption 並不必要。

## 小結

- 本講標題雖含「Alignment」，實際上幾乎全是 multimodality 的概覽；此處的「對齊」指影像—文字表示對齊與 VLM 投影對齊，不是偏好對齊。
- 多模態的第一原則是「一切皆 token」：把影像、影片、音訊都轉成 transformer 能吃的離散或連續 token，每個 token 承載一個語意單位。核心問題分成「怎麼輸入」與「怎麼生成」，本講主攻輸入。
- CLIP 用對比損失把成對的影像與文字向量拉近、非成對推遠，zero-shot 就能勝過在 120 萬張標註影像上訓練的 ResNet；但它需要大 batch、softmax 對整批運算而難以分解。
- SigLIP 把多類別對比改成逐對的 sigmoid 二元分類，讓 batch 與損失解耦、可平行、可用小 batch，效率更佳。
- VLM 的通用模板是「視覺編碼器 + 投影模組 + 語言模型」，先做 alignment phase 只訓投影模組，再分階段解凍；LLaVA 與 Qwen 兩家族都循此模板，差別在編碼器、投影模組與訓練階段的細節。
- AnyRes 與 dynamic resolution 把「影像任意解析度」化約成 transformer 早已擅長的「序列任意長度」問題；M-RoPE 則把位置編碼推廣到 (height, width, time) 三維。
- 這些 VLM 只在輸入側多模態、輸出永遠是文字；視覺編碼器遠小於語言模型，因為它只做局部 patch 操作、能力主要在語言模型。
- Chameleon 把影像用 VQ-VAE 離散化，讓單一語言模型統一處理並能生成影像，美學上優雅，但離散化丟細節、文字與影像熵差異導致訓練不穩，性能不如注入式 VLM。
- 影片帶來 token 爆量與 context length 壓力，需靠降幀、壓 patch、長 context、對長樣本做損失加權等手段平衡各模態權重。
- 目前 frontier omni-model 的細節不公開，講者推測是「連續編碼器 + diffusion 生成」；沒有單一通用編碼器，需求（語意 vs. 細節）不同，編碼器就不同。
