# Inference

## 導讀

前一章談的是「開始一次昂貴訓練之前，如何用小規模實驗預先知道大概會得到什麼」。這一章的視角完全不同：模型已經訓練好了，接下來呢？

答案是：接下來才是真正持續、每天都要付的成本。訓練是一次性的，再貴，做完就結束；但推論（inference）是重複成本，你每天都要付。講者用一個對比說明這件事的份量：OpenAI 估計每天生成 8.6 兆個 token，而同一年稍早發布的 GPT-4 訓練時總共只用了 32 兆個 token——也就是說，不到四天，單純為了服務使用者所產生的 token（以及背後對應的 compute），就已經追上訓練一個前沿模型所花的量。而且隨著語言模型從「聊天機器人」走向「agent」，這個對比只會更懸殊：agent 產生的大多數 token 根本不是給人讀的，而是思考、呼叫工具、自我檢查的過程，換句話說，生成的 token 數本質上就是花掉的 compute，而且沒有明顯上限。

這一章要處理三件事。第一，建立一套精確的數學語言，描述「推論為什麼慢、慢在哪裡」——答案幾乎都會指向同一個結構性原因：訓練可以在序列維度上平行處理所有 token，推論卻必須按照自回歸的順序一個一個生成。第二，介紹一整組工程手法，想辦法在不明顯犧牲準確度的前提下把推論做快——從縮小 KV cache、量化、剪枝，到一個特別優雅的無損技巧：speculative decoding。第三，把視角從「單一次生成」拉高到「服務一整個線上系統」，看系統設計者如何借用作業系統的老概念（分頁、碎片整理）來解決推論服務的動態批次問題。

## 核心內容

### 為什麼推論是一個獨立的工程問題

Transformer 訓練與推論用的是同一個模型、同一組權重，但工作負載的形狀完全不同。訓練時我們做 supervised fine-tuning 或語言模型的 next-token prediction，一次看到整個序列，可以把序列長度當成張量的一個維度，和 batch 維度一起塞進一次大矩陣乘法裡處理。推論不是這樣：自回歸生成要求你先產出第一個 token，才能產出第二個，再產出第三個——這是順序性的、無法在序列維度上平行化的過程。

這個差異聽起來像是常識，但它的後果很嚴重：如果不做任何最佳化，單純重複「把整個目前為止的序列丟進模型、取樣下一個 token、接上去、再丟一次」，每生成一個 token 都要花 `O(T²)` 的時間（因為 attention 本身是對序列長度的平方複雜度），生成 `T` 個 token 總共要花 `O(T³)`。這個立方級的代價，正是後面所有最佳化技巧要迴避的第一個敵人。

好消息是，只要 Transformer 是因果（causal）的，先前 token 的 key、value 表徵並不會因為你在後面接上新 token 而改變（如果是雙向模型就會變，那就沒有這個好消息了）。既然不會變，就沒有理由每次都重算——把它們存下來，重複利用就好。這就是 KV cache 的由來，也是這一章幾乎所有討論的起點：推論被拆成 prefill（一次性編碼整個 prompt、填滿 KV cache，和訓練一樣可以平行處理）與 generation（依序生成，每步只需要處理一個新 token，重複利用之前存好的 KV cache）兩個階段。

### 用 arithmetic intensity 量化「快」與「慢」

光說「推論比較慢」不夠精確。這一章延續 arithmetic intensity 的分析框架（flops 除以搬移的記憶體 bytes 數），把 prefill 與 generation、MLP 層與 attention 層分開算，結論非常清楚：

- MLP 層在 prefill 時的 arithmetic intensity 約正比於 `batch size × sequence length`；在 generation 時（一次只生成一個 token）約正比於 `batch size`（也就是同時服務的請求數）。只要能把足夠多的請求塞進同一個批次，MLP 這邊還算過得去。
- Attention 層則完全不同。它的 arithmetic intensity 因子是 `S·T/(S+T)`（`S` 是已有的 token 數，`T` 是要生成的 token 數）。Prefill 時（`T=S`）這個值約為 `S/2`，序列夠長就能維持高強度，沒問題；但 generation 時（`T=1`）這個值恆小於 1，遠低於現代加速器（例如 H100 大約需要 295 左右的強度才能打滿算力）。

這個落差不是巧合，而是有結構性原因：MLP 的權重對所有序列都一樣，批次越大，這些權重被分攤使用的次數越多，強度自然上升；但 attention 層裡，每個序列有自己專屬的 KV cache，這些資料量隨批次線性增加，並不會因為批次變大而被更有效率地共用。換句話說，增加批次大小幫不了 attention。這就是為什麼業界常說「推論是 memory-bound 的」——瓶頸不是算得不夠快，而是要不斷把記憶體（尤其是 KV cache）搬進搬出。

把這套分析套進一個具體模型（例如 Llama 2 13B 跑在 H100 上）可以看得更直觀：延遲（latency）大致是批次大小的一次函數（常數項是參數本身佔用的記憶體，係數是 KV cache 隨批次成長的速度）；吞吐量（throughput）則等於批次大小除以延遲，隨批次增加而改善，但邊際效益遞減，終究會被顯卡記憶體容量卡住，而不是真的趨近某個理論最大值。

### 一個真正的取捨：延遲與吞吐量

這裡帶出這一章最重要的工程直覺之一：延遲與吞吐量不是同一件事的兩種說法，而是實際上會互相拉扯的兩個目標。小批次的延遲較好、吞吐量較差；大批次的吞吐量較好、延遲較差。可以想像成搭公車：等公車坐滿才發車，個人等待時間（延遲）較長，但公車一次能載很多人，整體運輸效率（吞吐量）很好。

這對工程選擇有直接意義。如果你在做互動式應用（聊天機器人、agent 的即時回應），使用者在意的是「等第一個字要多久」（time to first token，本質上等於 prefill 花的時間）以及之後 token 出現的速度，這時應該傾向較小的批次。如果你在做批次資料處理（例如要用模型跑過一整批文件），你只在乎整個工作何時做完，這時應該把批次盡量拉大。沒有一個「最佳」批次大小，只有針對特定服務目標的取捨。

### 縮小 KV cache：一整套家族技巧

既然 KV cache 是 generation 階段記憶體壓力的主要來源，最直接的想法就是想辦法讓它變小。這一章介紹的手法形成一個清楚的譜系：

- **Grouped Query Attention（GQA）**把原本每個 head 各自的 key、value，改成幾個 group 共用——多頭注意力（`K=N`，不縮減）與 multi-query attention（`K=1`，幾乎沒人用，太犧牲品質）是兩個極端，GQA 選在中間找平衡。KV cache 縮小的倍率就是 `N/K`。
- **Multi-head Latent Attention（MLA）**，DeepSeek 提出，走的是不同的路：不減少 key、value 的「份數」，而是把產生 key、value 的表徵先壓縮到一個低維空間再存起來，需要時再還原。DeepSeek V2 把這個壓縮做得非常激進（從約一萬六千維壓到五百多維）。代價是這種壓縮和直接作用在 key、value 上的 RoPE 不相容，需要額外處理。
- **Cross-Layer Attention（CLA）**換一個維度共享：不是跨 head 共享，而是跨層共享——只有部分層真正計算 key、value，其餘層沿用前一層算好的結果。
- **Local / sliding window attention** 讓每個生成中的 token 只看最近一段固定窗口的歷史，KV cache 大小因此和序列長度脫鉤，對長 context 特別有利。但這是有代價的：窗口之外的資訊真的會被丟掉，是實打實犧牲表達力，因此實務上通常和全域（global）attention 交錯使用，只在部分層放寬到全域視野。
- **Linear attention**（以及 Mamba、DeltaNet 這類更精緻的版本）把整段歷史壓縮成一個固定大小的狀態，而不是保留完整的 KV cache。這類方法理論上更有彈性（遞迴機制某種程度上可以模擬 sliding window 的行為），但終究還是「沒有免費的午餐」——如果任務需要在極長歷史裡做精確檢索，把一切壓進固定大小的狀態必然會丟失資訊。

這些手法的排列方式提醒我們一件事：「壓縮」從來不是單一參數的選擇題，而是一整個設計空間——你可以壓 head 數量、壓表徵維度、壓層數共享、壓上下文窗口，或者換一種完全不同的狀態表示方式。而且沒有一個先驗答案能告訴你哪一種最適合你的模型，只能靠實驗找出「模型裡真正可以被擠壓的地方」。

### 系統層面的壓縮：量化與剪枝

量化（quantization）和剪枝（pruning）不是改架構，而是改「怎麼儲存、怎麼裁切一個已經訓練好的模型」。量化把數字的精度從 BF16 一路降到 INT4 甚至更低，原理很直接：更少的位元意味著更少的記憶體搬移，而推論是 memory-bound 的，記憶體變少直接等於變快。訓練時就考慮量化（quantization-aware training）能讓權重提前適應量化誤差，效果較好但成本高；訓練後才量化（post-training quantization）便宜得多，但需要一些技巧來補償誤差——例如利用 Hessian 資訊逐層量化並修正誤差（GPTQ），或者觀察到某些 activation channel 天生數值特別大、對應的權重更值得保留高精度（activation-aware quantization）。

剪枝則更粗暴：直接砍掉模型的一部分（某些隱藏單元甚至整層），砍完的模型會變差，再透過在目標資料上繼續訓練把它「治好」。Nvidia 的一個例子把 15B 模型剪成 8B，準確度沒有掉太多，而且修復所需的訓練量遠低於從頭訓練一個同樣大小的模型。這本質上是一種蒸餾：用一個好模型的部分權重，去初始化一個更快的架構，再修復它。

這些技巧共有的隱憂是：所有的「壓縮」都是有損的，必須用實際評測驗證準確度沒有掉太多——而且不同論文對同一種技巧的評價可能互相矛盾。提出 GQA 的論文顯示它效果不錯，但後來 DeepSeek 的比較顯示 GQA 其實傷害不小。這不代表其中一方錯了，而是提醒我們：任何「縮小規模但保持準確度」的宣稱，都值得用自己的評測再次確認。

### Speculative Decoding：一個無損的例外

前面所有手法都是有損壓縮，speculative decoding（推測解碼）則是這一章裡少見的無損技巧，而且思路相當優雅。核心觀察是：「檢查一段序列」比「生成一段序列」快得多。給定一段序列，平行地算出模型對每個位置的機率（就像 prefill 一樣），是又快又能打滿算力的操作；但要一個個生成，就慢得多。

利用這個不對稱性：先用一個便宜的小模型（draft model）一次生成幾個候選 token，再用真正要用的、昂貴的大模型（target model）平行地檢查這些候選——一次檢查一小批，而不是一個個生成。判定的規則是以機率 `min(1, Q/P)` 接受每個候選（`Q` 是 target 模型給的機率、`P` 是 draft 模型給的機率），一旦被拒絕就從一個修正過的殘差分佈裡取樣，結束這一輪。這套機制的巧妙之處在於：雖然形式上像是拒絕取樣（rejection sampling），但它保證每一輪都能拿到一個確實服從 target 模型分佈的樣本——不會像一般的拒絕取樣那樣「被拒絕就什麼都得不到」。

draft token 數太少，浪費了 target 模型的批次處理能力；太多，則拒絕率上升，划不來，存在一個經驗上的甜蜜點（原始論文的例子大約三、四個）。這也讓前面談的所有壓縮技巧有了第二個舞台：如果你把一個模型壓縮、蒸餾到很小，它未必要單獨拿去服務——即使它不夠格單獨上場，也可以拿來當 speculative decoding 的 draft 模型，讓大模型負責把關。

### 從單次生成到線上服務：動態批次與記憶體管理

前面所有分析都假設了乾淨、整齊的批次，但真實的服務場景完全不是這樣：請求隨時抵達，長度各不相同，有些請求還沒生成完、有些已經結束。這一章的最後，視角從「怎麼讓一次生成更快」切換到「怎麼讓一整個服務系統維持效率」。

第一個想法是 continuous batching（以 Orca 系統為代表）：不要等一整批請求都到齊、都算完才處理下一批，而是每一步為批次裡所有還在生成的序列各解碼一個 token，結束的序列隨時移出，新請求隨時可以插入——批次的組成持續在變。這帶來一個技術麻煩：一般的批次運算要求所有序列形狀一致，但不同長度的序列在 attention 計算上沒辦法直接共用張量（長度不同的 attention 矩陣形狀就是不一樣）。解法是把 attention 和 MLP 分開處理：attention 各算各的，MLP 因為對每個位置獨立運算，可以把所有序列串接成一個大序列一起算，這就是 selective batching。

第二個想法是 PagedAttention（出自 vLLM），處理的是「KV cache 該怎麼存」這個更底層的問題。因為你事先不知道一個請求會生成多長，如果按最大長度預留記憶體，大部分空間會浪費掉（內部碎片化）；不同請求之間留下的零碎空間也常常太小、用不上（外部碎片化）。解法幾乎是直接照搬作業系統虛擬記憶體分頁的老辦法：把 KV cache 切成固定大小、可以不連續存放的區塊，用索引表追蹤每塊實際存在哪裡。這個設計還帶來一個額外好處——共享前綴的請求（例如很多使用者共用同一個 system prompt，或同一個 prompt 生成多個不同的回應）可以直接共用同一批區塊的 KV cache，只在真正分岔的地方才複製、分裂（copy-on-write）。

系統工程師遇到「碎片化」問題就想起作業系統怎麼解決過同樣的問題，這件事本身也值得記住：很多推論服務的難題，不是全新的問題，而是老問題換了個場景重新出現。

## 從零實作角度或工程取捨

如果要從零實作一套可用的推論流程，這一章給出的清單相當具體：至少要正確實作 KV cache 的配置、讀寫與隨生成成長；要清楚劃分 prefill 與 generation 兩個階段，並理解為什麼前者可以平行、後者不行；要能在資源有限的情況下做出批次大小的選擇，並清楚意識到這個選擇本質上是在延遲與吞吐量之間做取捨，不存在放諸四海皆準的最佳值。

更深一層的工程心態是：遇到「讓推論變快」這個問題時，第一個該問的問題不是「怎麼減少計算量」，而是「怎麼減少要搬移的記憶體量，尤其是 KV cache」。這一章示範的所有架構層級手法（GQA、MLA、CLA、sliding window、linear attention）本質上都在回答同一個問題，只是切入的維度不同（head 數、表徵維度、層、上下文窗口、狀態表示）。系統層級的手法（量化、剪枝）則是在同一個目標下，選擇「事後裁切」而非「重新設計架構」。而 speculative decoding 提醒我們：並非所有加速手法都要犧牲準確度——只要能找到「檢查比生成快」這樣的結構性不對稱，就有機會做到無損加速。

最後，continuous batching 與 PagedAttention 提醒我們：一個推論系統的效率，不只取決於單次前向傳播算得多快，還取決於怎麼安排許多互相獨立、長度不一、抵達時間不同的請求共用有限的硬體資源。這已經超出「模型」的範疇，進入服務系統設計的範疇，但對於任何打算把語言模型真正服務起來的人，這部分和模型架構本身一樣重要。

## 常見誤解

第一個常見誤解是把「延遲」與「吞吐量」當成同一件事的兩種說法，認為讓推論變快就能同時改善兩者。多數改善記憶體使用效率的介入（例如縮小 KV cache）確實會同時改善兩者，但單純調整批次大小這個槓桿，延遲與吞吐量會朝相反方向移動——這是一個真正的取捨，不是工程上的疏漏。

第二個誤解是以為「模型比較小」就自動等於「推論比較快」。真正決定 generation 階段速度的，是 KV cache 的大小與記憶體頻寬，而不只是參數量；有些技巧（如 MLA）甚至保留了不少計算，只是把要儲存、搬移的表徵壓小了。

第三個誤解是以為所有壓縮技巧的效果可以互相佐證。這一章特別點出 GQA 論文與 MLA 論文對「GQA 是否會傷害準確度」給出了不一致的結論——同一種技巧在不同的模型、資料、評測方式下，結果可能不一樣，不能把單一論文的評測結果當成普遍真理。

第四個誤解是把 speculative decoding 誤認為一種「有點犧牲品質換取速度」的近似方法。它其實是精確、無損的——只要接受／拒絕的機率算對，取樣結果嚴格服從目標模型的分佈，速度提升來自於「檢查比生成快」這個結構性不對稱，而不是降低了品質。

第五個誤解是以為 Transformer 加上 KV cache 就是推論效率問題的終局解法。這一章結尾特別強調：KV cache 與 attention 的基本結構，某種程度上天生就對推論不友善；state space model、linear attention、diffusion model 這類從一開始就為推論設計的架構，可能提供比「在 Transformer 上打各種補丁」更根本的解法。

## 小結

- 推論與訓練是同一個模型的兩種截然不同的工作負載：訓練可以在序列維度上平行處理，推論的自回歸性質使其必須逐一生成，天生難以達到高 arithmetic intensity。
- KV cache 把樸素自回歸生成的 `O(T³)` 成本降下來，靠的是利用因果 Transformer 的性質：已經算過的 key、value 不會因為後面接上新 token 而改變。
- Prefill 階段是 compute-bound 的，generation 階段（尤其是 attention 部分）是 memory-bound 的——這正是「推論是 memory-bound」這個說法背後的精確原因。
- 延遲與吞吐量之間存在真正的取捨：小批次利於延遲，大批次利於吞吐量，選擇取決於服務目標是互動式應用還是批次處理。
- 縮小 KV cache 是這一章最大的一條主線，GQA、MLA、CLA、sliding window、linear attention 分別從不同維度（head 數、表徵維度、層、上下文窗口、狀態表示）進行壓縮，各有代價，都需要驗證準確度。
- 量化與剪枝是系統層面而非架構層面的壓縮手法，同樣遵循「縮小規模、驗證準確度」的原則。
- Speculative decoding 是少見的無損加速技巧，利用「檢查比生成快」的不對稱性，用小的 draft 模型搭配大的 target 模型，提供比逐一生成更快、但結果精確不變的方案。
- 真實的線上服務還要處理動態、長度不一的請求流，continuous batching 與 PagedAttention 借用作業系統的批次調度與分頁概念來解決效率與碎片化問題。
- 所有這些手法的共同原則是：在不明顯犧牲準確度的前提下，減少推論要搬移的記憶體量（尤其是 KV cache），因為推論的瓶頸幾乎總是記憶體，而不是算力。
- 更長遠來看，Transformer 加 KV cache 的組合可能不是推論效率問題的終極答案；state space model、linear attention、diffusion model 等從設計之初就考慮推論友善性的新架構，可能帶來更根本性的改善。
