# Lecture 5 Reading Notes: Asymptotic Equipartition Property (AEP)

## Overview
Lecture 5 covers the Asymptotic Equipartition Property (AEP) and its deep connections to data compression. It introduces the mathematical foundations showing that for long sequences generated by an IID source, the probability mass is concentrated in a tiny subset of all possible sequences, called the "typical set".

## Key Concepts
- **Asymptotic Equipartition Property (AEP)**: Based on the Weak Law of Large Numbers, the empirical average of $- \log P(U_i)$ converges to its expected value, which is the entropy $H(U)$.
- **The $\epsilon$-Typical Set $A_\epsilon^{(n)}$**:
  - The set of sequences whose empirical entropy is close to the true entropy: $2^{-n(H(U)+\epsilon)} \le P(U^n) \le 2^{-n(H(U)-\epsilon)}$.
  - **Properties**:
    1. Probability of the typical set is close to 1 as $n \to \infty$.
    2. Size of the typical set is approximately $2^{nH(U)}$.
    3. The typical set is the smallest set containing most of the probability mass. Any set smaller than $2^{n(H(U)-\delta)}$ has negligible probability.
- **Compression using Typical Sets**:
  - Since there are roughly $2^{nH(U)}$ typical sequences, we can index them using $nH(U)$ bits.
  - Atypical sequences are rare, so encoding them with a wasteful scheme (e.g., raw bits) does not significantly impact the expected code length.
  - This guarantees we can achieve an expected code length of $H(U) + \epsilon$ bits per symbol.
- **Mismatched Compression**:
  - If we design a code assuming a probability distribution $Q$, but the true distribution is $P$, the expected code length is $H(P) + D_{KL}(P || Q)$.
  - The penalty for the mismatch is exactly the Kullback-Leibler (KL) Divergence $D_{KL}(P || Q)$.
- **Experimental Results & Complexity**:
  - Huffman, Shannon, and Typical Set Coders all perform close to the entropy bound for reasonable block sizes.
  - However, they all suffer from exponential complexity (time and space) as the block size $n$ increases. This motivates the need for practical algorithms like Arithmetic Coding.
- **Canonical Huffman Codes**:
  - Huffman codes are not strictly unique. By imposing rules (sorted by length, then lexicographically), we get a Canonical Huffman Code.
  - Only the code lengths need to be stored/transmitted, drastically saving overhead (used in gzip/deflate).

## Additional Notes
- Optimal code implies optimal expected length, not optimal for every specific sequence realization.
